# FE 스펙 작성 스킬

러프한 초안이든 빈 상태든, 구조화된 질문으로 빠진 부분을 찾고 Phase 기반 정제 스펙을 생성한다.

## 목적

- 러프 초안 → 구조화된 질문 → Phase 기반 정제 스펙
- "이 스펙대로 Phase별로 만들어라"로 위임 가능한 문서 생성
- 페이지 내 모달/드로어 등 하위 UI도 구조적으로 포함

## 입력

- `$ARGUMENTS`: 페이지명 또는 기능명 (예: `주문 상세`, `서류 등록 모달`)
- 디자인 참조: 사용자가 별도 제공 (스크린샷, 텍스트 설명 등)
- 러프 초안: 사용자가 이미 작성한 비정형 스펙 (있으면)

## 진행 단계

### Phase 1: 정보 수집

**입력 분석:**

| 입력 유형 | 처리 |
|-----------|------|
| 러프 초안 있음 | 읽고 정보 추출, 부족한 부분만 추가 질문 |
| 스펙 경로 제공 | 해당 문서 Read 후 정보 추출 |
| 없음 | AskUserQuestion으로 순차 질문 |

**AskUserQuestion으로 질문 (초안 없을 때, 최대 2회로 압축):**

1회차 AskUserQuestion (최대 4개 질문 묶음):
- "어떤 페이지/기능의 스펙을 작성할까요?" (ARGUMENTS 없을 때만)
- "URL 경로 / 주요 기능 / 이 기능의 성공 기준은?"
- "API 참조 방식은?" → "백엔드 코드 경로" / "swagger" / "API 스펙 문서 이미 있음" / "아직 없음"
- "디자인 참조 또는 참고할 유사 페이지가 있나요?"

2회차는 1회차 답변 기반으로 부족한 부분만 추가 질문 (필요 없으면 스킵)

### Phase 2: 기존 코드/스펙 확인

1. `.ai/specs/` 폴더 확인 — 관련 스펙 이미 있는지
2. `src/pages/` 에서 유사 페이지 구조 샘플링 (참고할 페이지가 있으면 해당 페이지)
3. API 참조 방식에 따라:
   - 백엔드 코드 → controller/DTO 분석
   - swagger → 엔드포인트 파악
   - `.ai/specs/api/` → 기존 스펙 읽기

### Phase 3: Gap Analysis (구조화된 질문)

수집한 정보와 코드를 기반으로, **빠진 부분과 모호한 점을 능동적으로 찾아 질문한다.**

**3-1. 빠진 요구사항 탐색:**
- 수집 정보에서 누락 가능성이 있는 항목을 AI가 식별
- AskUserQuestion (multiSelect): "다음 중 이 기능에 포함되어야 할 항목이 있나요?"
  - 식별된 누락 후보들을 선택지로 제시
  - 예: "빈 데이터 상태 처리", "권한별 UI 분기", "에러 복구 UI", "낙관적 업데이트" 등
- 누락 후보가 없으면 스킵

**3-2. 모호한 지점 명확화:**
- 스펙에서 해석이 갈리는 부분을 식별
- AskUserQuestion: 구체적 양자택일 질문
  - 예: "상태 변경 실패 시 이전 상태로 롤백하나요, 에러 메시지만 표시하나요?"
  - 예: "리스트가 빈 경우 빈 테이블을 보여주나요, 안내 문구를 보여주나요?"
- 모호한 지점 없으면 스킵

**3-3. 컨텍스트 충돌 확인:**
- Phase 2에서 파악한 기존 코드 패턴과 다른 점이 있으면 질문
- AskUserQuestion: "기존 {페이지}는 {패턴}을 쓰는데, 이 페이지도 동일하게 할까요?"
- 충돌 없으면 스킵

**규칙:**
- 각 카테고리에서 질문이 없으면 해당 단계 스킵 (불필요한 질문 금지)
- 질문은 AskUserQuestion **최대 2회**로 압축 (질문 폭탄 금지)
- 사용자 답변을 스펙에 반영

### Phase 4: 스펙 문서 생성

**위치**: `.ai/specs/{page-name}.md`

**템플릿:**

```markdown
# {페이지명} 스펙

> 작성일: {date}
> 상태: draft | review | approved
> URL: {route path}

## 개요

{이 페이지가 무엇을 하는지 1-2문장}

## 페이지 구조

{PageName}Page
├── {SectionA}        ← 설명
│   ├── {Component1}  ← 설명
│   └── {Component2}  ← 설명
├── {SectionB}        ← 설명
└── [Modal] {ModalName} ← 트리거: {버튼/조건}

## 섹션별 상세

### {SectionA}

**UI 요소:**
| 요소 | 타입 | 설명 | 비고 |
|------|------|------|------|

**동작:**
- {인터랙션 설명}

**API 매핑:**
- 조회: `GET /api/...` → `fetch{Resource}` → `{resource}Query.list()`

### [Modal] {ModalName}

> 트리거: {어떤 버튼/조건으로 열리는지}
> 닫기: {저장 후 자동닫기 / X 버튼 / 외부 클릭}

**UI 요소:**
| 요소 | 타입 | 설명 | 비고 |

**폼 스키마:** (폼이 있을 때)
const {modalName}Schema = z.object({ ... });

**동작:**
- 제출 시: {API 호출} → {성공 시 동작}
- 에러 시: {에러 토스트}

## API 요약

| Method | Endpoint | Remote 함수 | 용도 |
|--------|----------|-------------|------|

> API 상세는 `.ai/specs/api/{domain}.md` 참조

## 파일 구조 (생성 예정)

src/pages/{domain}/{page}/
├── {Page}Page.tsx
├── models/{resource}.dto.ts
├── remotes/{resource}.ts
├── queries/{resource}.query.ts
├── mutations/{resource}.mutation.ts
├── types/{resource}.schema.ts      ← 폼 있을 때
└── components/

## 엣지 케이스

- {빈 상태 처리}
- {권한 없을 때}
- {로딩/에러 상태}

## 구현 Phases

### Phase 1: {제목} (예: API 계층 구축)
- [ ] {구현 항목}
- [ ] {구현 항목}
- 검증: {완료 기준 — 예: 타입체크 통과}

### Phase 2: {제목} (예: 페이지 레이아웃 + 컴포넌트)
- [ ] {구현 항목}
- 검증: {완료 기준 — 예: 화면 렌더링 확인}

### Phase 3: {제목} (예: 인터랙션 + 엣지 케이스)
- [ ] {구현 항목}
- 검증: {완료 기준 — 예: 전체 시나리오 동작 확인}
```

**Phase 분리 기준:**
- Phase 간 의존성 최소화 — 각 Phase가 독립적으로 검증 가능
- Phase 1은 항상 데이터 계층 (DTO/Remote/Query/Mutation)
- 중간 Phase는 UI 구조 + 컴포넌트
- 마지막 Phase는 인터랙션 연결 + 엣지 케이스
- Phase 수는 기능 복잡도에 따라 2~5개

### Phase 5: 검토 요청

1. 생성된 스펙 문서를 사용자에게 보여줌
2. AskUserQuestion: "스펙을 검토해주세요. 수정할 부분이 있나요?"
   - 수정 요청 → 반영 후 다시 검토
   - 승인 → 상태를 `approved`로 변경

## 출력 형식

```
스펙 문서 생성 완료: .ai/specs/{page-name}.md

포함된 내용:
- 페이지 구조: {섹션 수}개 섹션, {모달 수}개 모달
- API: {엔드포인트 수}개 엔드포인트
- 구현 Phases: {Phase 수}개 Phase
- Gap Analysis: {질문 수}개 질문으로 {반영 수}개 항목 보완

다음 단계: 스펙 승인 후 Phase별 구현 시작
```

## 모달/드로어 처리 규칙

- 페이지 스펙 안에 `### [Modal] 모달명` 또는 `### [Drawer] 드로어명` 섹션으로 포함
- 각 모달/드로어는 독립 섹션: 트리거 조건, UI 요소, 동작, API 매핑을 모두 기술
- 컴포넌트 파일은 `components/{Name}Modal.tsx` 또는 `components/{Name}Drawer.tsx`로 분리
- 복잡한 모달(3개+ 섹션)은 별도 스펙 파일로 분리 가능

## 스펙 품질 체크리스트

스펙이 아래 기준을 만족하면 AI가 Phase별로 구현 가능:

- [ ] 모든 UI 요소가 타입(Input/Select/Table 등)과 함께 나열됨
- [ ] 모든 인터랙션에 "트리거 → 동작 → 결과"가 명시됨
- [ ] 모든 API 호출이 엔드포인트 + Remote 함수명으로 매핑됨
- [ ] 폼이 있으면 Zod 스키마가 포함됨
- [ ] 모달/드로어의 열기/닫기 조건이 명시됨
- [ ] 파일 구조(생성 예정 파일 목록)가 포함됨
- [ ] 엣지 케이스(빈 상태, 에러, 권한)가 포함됨
- [ ] 구현 Phases가 검증 기준과 함께 정의됨
- [ ] Gap Analysis 결과가 스펙에 반영됨
