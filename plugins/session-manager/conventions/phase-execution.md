# Phase 기반 실행 규칙

스펙에 `## 구현 Phases` 또는 `## Phases`가 있으면, 반드시 Phase별로 끊어서 실행한다.

## 규칙

### 1. 한 번에 하나의 Phase만 구현

- 현재 Phase의 모든 항목을 완료한 후 다음으로 넘어감
- Phase를 건너뛰거나 합치지 않음
- "간단하니까 합쳐서 하겠습니다" 금지 — Phase 분리는 검증을 위한 것

### 2. Phase 완료 시 구조화된 검증

Phase를 완료하면 **단순 "됐어?" 가 아니라, 구체적 검증 포인트를 보고한다.**

**보고 형식 (정본 — implement 등 커맨드에서 이 형식을 참조):**

```
Phase {N} 완료: {제목}

구현 내용:
- {완료 항목 1}
- {완료 항목 2}

검증 결과:
- {스펙 검증 기준}: {통과/실패 + 근거}

확인 사항:
1. 컨텍스트 적합성: {기존 패턴과 일관적인지 AI 판단 + 근거}
2. 스펙 차이: {스펙과 다르게 구현한 부분이 있으면 사유 설명}
3. 다음 Phase 영향: {이번 결정이 다음 Phase에 영향 주는 부분}

결정사항: (이번 Phase에서 내린 판단이 있으면)
- {결정 + 근거}
```

이후 AskUserQuestion으로 사용자 확인:
- "Phase {N} 결과를 확인해주세요."
- 선택지: "다음 Phase 진행" / "수정 필요" / "스펙 변경 필요" / "나머지 전부 진행"
- "나머지 전부 진행" 선택 시 → 남은 Phase 연속 실행 (사용자 확인 생략, active 파일 업데이트는 계속 수행)

### 3. active 파일 업데이트

- Phase 완료 시 active 파일의 체크리스트 `[x]` 업데이트
- 결정사항이 있으면 `## 결정사항` 섹션에 기록 (없으면 생성)
- 컨벤션 변경이 있으면 `## 컨벤션 변경` 섹션에 기록 (없으면 생성)
- 스펙과 달라진 부분이 있으면 `## 스펙 변경` 섹션에 기록 (없으면 생성)

> 이 기록들은 `/done` 시 자가학습 소스로 사용됨 (done.md Step 2-B)

### 4. 스펙 변경 시 명시

구현 중 스펙과 달라져야 하는 부분이 발견되면:
- 즉시 사용자에게 알림 (자의적 변경 금지)
- 승인 받으면 스펙 문서와 active 파일 모두 업데이트

## 적용 범위

- `.ai/specs/` 내 스펙에 `## 구현 Phases` 또는 `## Phases`가 정의된 작업
- `/fe:implement`, 직접 구현 요청 등 구현 작업 전반
- Phase가 없는 스펙이면 이 규칙은 적용하지 않음
- 버그 수정, 단순 변경 등 스펙 없이 진행하는 작업에는 적용하지 않음

## 강제 메커니즘

Convention만으로는 AI가 무시할 수 있다. 실제 강제는 다음으로 보장:
- `/fe:implement` 커맨드가 Phase 감지 시 Phase별 Agent 위임을 내장 (Phase 2A)
- 직접 구현 요청 시에도 이 convention이 컨텍스트에 로드되어 있으면 Phase 감지 후 순차 실행
