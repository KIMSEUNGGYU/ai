대화 내용과 외부 자료를 분석하여 구조화된 마크다운 문서를 작성하고, 적절한 위치에 저장한다.

`/note`와의 차이: `/note`는 세션 중 떠오른 짧은 지식을 빠르게 메모하는 것이고, `/doc`은 대화 맥락과 외부 자료를 종합하여 완성된 문서를 만드는 것이다. 슬랙 스레드 정리, 트러블슈팅 기록, 설정 가이드, 의사결정 기록 등 "문서"라고 부를 수 있는 결과물이 필요할 때 사용한다.

## 동작 흐름

### 1단계: 자료 수집

사용자가 제공한 정보를 파악한다:

- **대화 내용**: 현재 세션에서 논의된 내용
- **외부 자료**: 사용자가 언급한 URL이나 파일

외부 자료가 있으면 적절한 도구로 읽는다:

| 소스 | 도구 | 예시 |
|------|------|------|
| Slack 스레드 | `slack_read_thread` | 슬랙 URL에서 channel_id, message_ts 추출 |
| Notion 페이지 | `notion-fetch` | 노션 URL로 페이지 내용 조회 |
| Linear 이슈 | `get_issue` | 이슈 ID로 컨텍스트 조회 |
| 웹 페이지 | `WebFetch` | 공식 문서, 블로그 등 |
| 로컬 파일 | `Read` | 코드, 설정 파일, 로그 등 |

추가 자료가 필요해 보이면 사용자에게 물어본다:
> "관련 슬랙 스레드나 참고 자료가 있으면 URL을 알려주세요. 없으면 대화 내용만으로 정리합니다."

### 2단계: 문서 유형 판별

수집된 내용을 분석하여 문서 유형을 자동 판별한다. 사용자가 명시하면 그대로 따른다.

| 유형 | 판별 신호 |
|------|----------|
| `troubleshooting` | 에러, 버그, 장애, 해결 과정이 포함 |
| `guide` | 설정 방법, 구축 절차, 사용법 설명 |
| `decision` | 선택지 비교, A vs B, 결정 근거 |
| `discussion` | 여러 사람의 의견, 슬랙 스레드 논의 |

판별이 애매하면 사용자에게 확인한다.

### 3단계: 문서 작성

#### Obsidian frontmatter

모든 문서에 Obsidian 호환 frontmatter를 포함한다:

```yaml
---
tags:
  - {도메인 태그}
  - {기술 태그}
date: YYYY-MM-DD
type: troubleshooting | guide | decision | discussion
source:
  - {원본 자료 URL 또는 경로}
aliases:
  - {검색/링크용 별칭}
---
```

- `tags`: 내용에서 핵심 키워드 2~4개 추출 (한글 허용)
- `date`: 문서 작성일 (오늘 날짜)
- `type`: 판별된 문서 유형
- `source`: 원본 자료의 출처 (슬랙 URL, 노션 URL, 파일 경로 등)
- `aliases`: 제목의 축약형이나 다른 표현 (Obsidian 검색에 유용)

#### 유형별 템플릿

**troubleshooting** — 문제 해결 과정 기록

```markdown
# {제목}

> {날짜} | {한 줄 요약}

## 발단
{문제가 발생한 상황, 증상}

## 원인
{근본 원인 분석}

## 조치
{해결 과정, 단계별}

## 교훈
{재발 방지, 향후 참고사항}
```

**guide** — 절차/방법 안내

```markdown
# {제목}

> {한 줄 요약}

## 개요
{무엇을, 왜 하는지}

## 절차
{단계별 진행 방법}

## 주의사항
{함정, 흔한 실수, 참고할 점}
```

**decision** — 의사결정 기록

```markdown
# {제목}

> {날짜} | {결정 한 줄 요약}

## 배경
{왜 이 결정이 필요했는지}

## 선택지
{검토한 대안들과 각각의 장단점}

## 결정
{최종 선택}

## 근거
{왜 이것을 선택했는지}
```

**discussion** — 논의/회의 기록

```markdown
# {제목}

> {날짜}

## 참여자
{논의 참여자}

## 논의 내용
{주요 논점별 정리}

## 결론
{합의된 내용}

## 액션 아이템
{누가 무엇을 할 것인지}
```

### 4단계: 초안 제시 및 확인

문서를 작성하기 전에 **어떤 형태로 만들지** 먼저 보여주고 확인받는다:

```
📄 문서 초안
  유형: troubleshooting
  제목: DX ECS 외부 접근 보안 이슈
  구조: 발단 → 원인 → 조치 → 교훈
  포함할 Q&A:
    - Q. HOSTNAME=0.0.0.0은 바꿔야 하나?
    - Q. ALB SG로 제한하면 안전한 거지?
  저장 위치: ~/hq/10_Work/dx/dx-ecs-security-incident-2026-03-18.md

이대로 작성할까요?
```

사용자가 유형, 구조, 포함 항목, 저장 위치를 조정할 수 있다. 확인받은 후 본문을 작성한다.

### 5단계: 저장 위치 결정

`pc-map.md` 규칙에 따라 저장 위치를 제안한다:

| 내용 성격 | 저장 위치 | 예시 |
|----------|----------|------|
| 특정 서비스 업무 | `~/hq/10_Work/{서비스}/` | DX 트러블슈팅, admin 설정 가이드 |
| 학습/개념 정리 | `~/hq/20_Learn/` | React 패턴, AWS 개념 |
| 프로젝트 작업 문서 | `.ai/notes/` | 현재 프로젝트 한정 내용 |
| 아이디어 | `~/hq/05_Ideas/` | 개선 아이디어, 실험 |
| 시스템/설정 | `~/hq/40_System/` | 개발 환경, 도구 설정 |
| 분류 불명확 | `~/hq/00_Inbox/` | 나중에 정리 |

파일명은 `{주제}-{날짜 또는 키워드}.md` 형식으로 제안한다.
기존 파일이 있는지 Glob으로 확인하고, 있으면 업데이트를 제안한다.

### 6단계: 저장 및 완료

저장 후 결과를 출력한다:

```
/doc 저장 완료
  문서: {제목}
  유형: {type}
  위치: {저장 경로}
```

## 작성 원칙

- 대화 원문을 그대로 복붙하지 않는다 — 핵심을 추출하여 구조화
- **사용자의 질문은 반드시 포함한다** — 대화 중 사용자가 던진 질문과 그에 대한 답변은 문서의 이해를 돕는 핵심 맥락이다. `Q.` 키워드로 질문을 표시하고 바로 아래에 답변을 배치한다. 질문이 곧 "이 문서를 읽는 미래의 나도 궁금할 것"이기 때문. 예: `> Q. HOSTNAME=0.0.0.0은 바꿔야 하나?`
- 슬랙 대화는 참여자별 발언을 정리하되, 의미 없는 리액션/인사는 생략
- 코드는 핵심 부분만 포함, 전체 복붙 금지
- 이미지 첨부가 슬랙에 있었으면 `![[Pasted image ...]]` 자리 표시 (사용자가 Obsidian에서 직접 첨부)
- source에 원본 URL을 반드시 기록하여 출처 추적 가능하게
