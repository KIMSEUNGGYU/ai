# Agent Harness 치트시트

## 5축 한눈에

```
         Tools (능력)            → tools: ["Read", "Glob"]
            │                      tools 생략 = 전부 허용
            │                      tools: [] = 전부 차단 (MCP 제외)
Hooks ──────┼────── Subagents
(감시)       │       (위임)
Pre: 차단   │       agents: { "name": { prompt, tools } }
Post: 관찰  │       내부적으로 Task 도구로 호출
            │       → tools에 Task 없으면 서브에이전트 불가!
   Prompt ──┼─── Sessions (기억)
   (성격)          resume: sessionId
   역할+관점+형식  대화 맥락만 유지, 옵션은 리셋
```

## SDK 옵션 매핑

| 축 | SDK 옵션 | 플러그인 대응 | 핵심 |
|---|----------|-------------|------|
| 루프 | `query()` | Claude Code 내장 | 도구 호출 없으면 종료 |
| 도구 | `tools` | 없음 (자동) | 생략≠[], MCP는 독립 레이어 |
| 프롬프트 | `systemPrompt` | agents/*.md 본문 | 가이드이지 템플릿 아님 |
| 훅 | `hooks` | hooks/*.json | SDK=TS함수, 플러그인=셸 |
| 서브에이전트 | `agents` | agents/*.md 선언적 | 컨텍스트 분리가 핵심 가치 |
| 세션 | `resume` | 없음 (자동) | 기억만 유지, 설정은 매번 재지정 |
| MCP | `mcpServers` | .mcp.json | Tools의 확장점, 추가만 가능 |

## 하네스 고유 패턴: 평가 루프

```typescript
let result = await runAgent(task);
while (!evaluate(result)) {           // 코드가 판정 (결정론적)
  result = await runAgent(task, {      // resume으로 피드백
    feedback: getErrors(result)
  });
}
```

- 프롬프트 설계의 불완전성을 **런타임에 보정**
- 플러그인(1회 실행)으로는 불가능

## 자주 빠지는 함정

| 함정 | 정답 |
|------|------|
| tools: ["Read"] 했는데 서브에이전트 안 됨 | Task가 리스트에 없어서. tools 생략 필요 |
| PreToolUse로 Read 차단했는데 파일 읽힘 | MCP 등 대체 경로로 **우회**. 모든 경로 차단 필요 |
| resume 했는데 도구가 안 됨 | 옵션 리셋됨. tools/hooks/agents 재지정 필요 |
| 평가 1회 통과 → 루프 불필요? | 안전망 역할. 프롬프트가 좋으면 발동 안 하는 게 이상적 |
| 피드백 줬더니 결과가 폭발적으로 김 | LLM 과보정. 피드백에 "누락만 추가" 명시 or 상한 체크 |

## 핵심 한 줄

> **플러그인 = 남의 하네스에 부품 꽂기 / SDK = 하네스를 직접 만들기**
> 차이는 "루프 제어" — 결과를 검증하고, 피드백하고, 재시도하는 프로그래밍 가능한 루프.
