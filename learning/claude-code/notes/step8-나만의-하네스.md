# Step 8: 나만의 하네스 — 전체 통합

## 배운 것

### 이중 루프 구조
- **내부 루프**: SDK의 `query()` — Claude가 도구를 쓰면서 작업하는 에이전트 루프
- **외부 루프**: TypeScript의 `for` — 결과가 기준을 통과할 때까지 반복하는 평가 루프
- 내부는 SDK가 제어, 외부는 **우리 코드**가 제어 — 이것이 "하네스"의 본질

### 5축 통합 경험
| 축 | SDK 옵션 | 이 시스템에서의 역할 |
|---|----------|---------------------|
| 루프 | `query()` + `for` | 에이전트 루프 + 평가 루프 이중 구조 |
| 도구 | tools 생략 (전체 허용) | Task 포함 → 서브에이전트 호출 가능 |
| 프롬프트 | `systemPrompt` | 오케스트레이터 역할 ("반드시 포함" 지시) |
| 훅 | `PostToolUse` | 전체 도구 사용 audit.log 기록 |
| 서브에이전트 | `agents.file-reviewer` | 파일별 코드 리뷰 위임 |
| 세션 | `resume` | 평가 실패 → 피드백 전달 → 재시도 |

### 실행 결과 (2회 실행)

**1차 실행 (관대한 기준)**
- 평가 1회 만에 통과
- systemPrompt에 "반드시 포함" 지시가 잘 작동
- 비용: $0.30, 6턴

**2차 실행 (강화된 기준: 보안 분석 + 라인 번호)**
- 시도 1: 코드블록 8개, 심각도 ✓, 제안 ✓, **라인번호 ✗, 보안 ✗** → 실패
- 피드백: "구체적 라인 번호 참조, 보안 취약점 분석 누락"
- 시도 2: resume + 피드백 → **코드블록 50개**, 라인번호 ✓, 보안 ✓ → 통과
- 과보정 현상: 피드백에 과잉 반응하여 리뷰가 훨씬 상세해짐

### 설계 결정과 그 이유

1. **tools 생략 (vs tools 명시)**
   - Step 5 교훈: `tools: ["Read", "Glob"]` 하면 Task가 없어서 서브에이전트 호출 불가
   - `tools` 생략 = 전부 허용 (Task 포함)

2. **평가 함수 = TypeScript (vs 서브에이전트)**
   - 에이전트(LLM) = 비결정적 → 평가 기준이 매번 달라질 수 있음
   - 코드 함수 = 결정론적 → 기준이 명확하고 예측 가능
   - 하네스의 가치: "LLM의 창의성 + 코드의 엄밀함"

3. **공통 옵션 변수 (baseOptions)**
   - Step 6 교훈: `resume`은 대화 맥락만 이어가고, 옵션은 리셋
   - resume 시에도 tools, agents, hooks 재지정 필요

## 핵심 인사이트

> **평가 루프의 진정한 가치는 "프롬프트 설계의 불완전성을 런타임에 보정"하는 것.**
>
> systemPrompt에 "보안 분석"을 깜빡했어도, evaluateReview()가 "보안 분석 누락"을
> 감지하고 피드백으로 전달하면 에이전트가 보완한다.
>
> 프롬프트 엔지니어링만으로 완벽할 수 없다 — 평가 루프는 이 한계를 구조적으로 해결한다.
> 이것이 플러그인(1회 실행)으로는 불가능하고, 하네스(프로그래밍 가능한 루프)만의 고유 가치.

## 관찰: 피드백 과보정

- 1차: 코드블록 8개 → 2차: 코드블록 50개
- LLM은 "부족하다"는 피드백에 과잉 반응하는 경향
- 실전에서는 평가 함수에 상한 체크도 필요할 수 있음 (너무 길면 거절)

## 관찰: 서브에이전트 병렬 실행

- audit.log 타임스탬프: Read 4건이 거의 동시(35~38초) → 병렬 실행 증거
- Task 완료 기록은 나중(52~76초) → PostToolUse는 완료 후 실행
- 메인 에이전트가 `Task` 4개를 한 번에 호출 → SDK가 병렬 처리

## 하네스 5축 최종 정리

```
하네스 = 에이전트 루프를 제어하는 실행 환경

┌──────────────────────────────────────┐
│           query() — 에이전트 루프       │
│                                      │
│  systemPrompt ── 역할과 관점 정의      │
│  tools ── 사용 가능한 능력             │
│  agents ── 작업 위임 (서브에이전트)     │
│  hooks ── 관찰 + 개입 (감사/차단)      │
│  resume ── 기억 유지 (세션)            │
│                                      │
│  + 외부 평가 루프 ── 코드로 제어        │
│    (플러그인으로는 불가능한 영역)        │
└──────────────────────────────────────┘
```

## 플러그인 vs SDK 최종 비교

| | 플러그인 | SDK (하네스) |
|---|---------|-------------|
| 루프 제어 | ✗ 불가 | ✓ 평가 루프, 조건부 재시도 |
| 도구 제어 | 자동 | ✓ 정밀 화이트리스트 |
| 서브에이전트 | 선언적 (.md) | ✓ 프로그래밍적 + 병렬 |
| 훅 | 셸 명령어 | ✓ TypeScript 함수 (async, 상태) |
| 세션 | 자동 관리 | ✓ 프로그래밍적 resume/fork |
| 배포 | Claude Code 안에서만 | ✓ CI/CD, 서버, 크론잡 |
