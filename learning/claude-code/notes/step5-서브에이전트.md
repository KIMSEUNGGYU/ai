# Step 5: Subagents — 병렬 코드 분석

## 배운 것

- `agents` 옵션으로 커스텀 서브에이전트 정의 (description, prompt, tools)
- 서브에이전트는 내부적으로 `Task`라는 도구로 호출됨
- `description`이 핵심 — 메인 에이전트가 이걸 읽고 호출 여부 판단

## 핵심 인사이트

### tools 제한이 서브에이전트도 막는다

1차 실행에서 `tools: ["Read", "Glob", "Grep"]`으로 제한했더니 Task 도구가 화이트리스트에 없어서 서브에이전트 호출 불가. Claude가 "접근할 수 없지만 직접 하겠습니다"라고 우회.

| `tools` 설정 | 도구 수 | 서브에이전트 |
|---|---|---|
| 생략 (안 씀) = 전부 허용 | 90개 | 호출 가능 |
| `["Read", "Glob", "Grep"]` = 화이트리스트 | 72개 | 호출 불가 (Task 없음) |
| `[]` = 빌트인 전부 차단 | MCP만 | 호출 불가 |

주의: `tools` 생략 ≠ `tools: []`. 생략은 전부 허용, `[]`는 전부 차단.

### user 메시지를 까보면 서브에이전트 동작이 보인다

user 메시지에 tool_result 내용이 들어있어서, 출력하면 서브에이전트의 전체 작업 과정이 보임:

```
🤖 [9]  tools: [Task]     ← 메인이 서브에이전트 호출
👤 [10] "파일을 리뷰해주세요..."  ← 서브에이전트에게 보낸 프롬프트
🤖 [11~20] ...             ← 서브에이전트 내부 도구 호출 과정
👤 [21] "상세한 리뷰 분석..."    ← ⭐ 서브에이전트 최종 결과 (Task의 tool_result)
🤖 [23] "완료되었습니다!"  ← 메인이 결과 종합
```

### 메인 vs 서브에이전트 구분은 흐름으로 추론

메시지 자체에 "서브에이전트 메시지입니다"라는 표시는 없음.
Task 호출(`[9]`) ~ Task 결과(`[21]`) 사이가 서브에이전트 구간이라고 흐름으로 추론하는 수밖에 없음.

### Step 3(systemPrompt)과의 비교

| | Step 3 (systemPrompt) | Step 5 (subagent) |
|---|---|---|
| 누가 리뷰 | 메인 에이전트 자신 | 서브에이전트 |
| 컨텍스트 | 메인 대화에 쌓임 | 독립 (메인은 결과만 받음) |
| 메인의 역할 | 직접 분석 | 결과 종합/전달 |

서브에이전트의 실용적 가치 = 컨텍스트 분리. 리뷰 상세가 메인 대화를 오염시키지 않음.
