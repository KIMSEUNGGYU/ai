# 세션 관리와 작업 연속성

> **핵심 질문:** "어떻게 하면 내가 했던 작업을 잘 이어서 할 수 있을까?"

## Claude Code 세션의 실체

### 저장 구조

```
~/.claude/projects/<project-hash>/
├── <session-id>.jsonl          ← 대화 전체 기록 (append-only)
├── <session-id>/               ← 세션 부속 파일 (있는 세션만)
│   ├── subagents/              ← 서브에이전트 대화 기록
│   └── tool-results/           ← 큰 도구 결과물
└── ...
```

### JSONL 메시지 타입 분포 (실제 측정)

```
 67x  progress/hook_progress     ← 훅 실행 상태
 36x  progress/bash_progress     ← bash 명령 진행
 33x  assistant                  ← Claude 응답
 26x  user                       ← 사용자 입력 + tool_result
  8x  file-history-snapshot      ← 파일 변경 스냅샷
  5x  system/stop_hook_summary   ← 종료 훅
  3x  system/turn_duration       ← 턴 소요시간
```

JSONL = 대화의 raw log. 텍스트뿐 아니라 도구 호출, 훅 진행, 파일 스냅샷까지 전부 포함.

### session_id

- `system/init` 메시지에서 **딱 한 번** 발행됨
- 이후 assistant, result 메시지에는 없음
- SDK에서는 `resume: sessionId`로 이전 대화 이어가기 가능
- `forkSession: true`로 같은 지점에서 분기(Git branch처럼) 가능

## 기억의 3개 레이어

```
Layer 1: JSONL 파일 (세션 전체 기록)
  → 모든 것이 기록됨, 절대 삭제 안 됨
  → "녹음 테이프"

Layer 2: 컨텍스트 윈도우 (Claude가 실제로 보는 것)
  → ~200K 토큰 제한 → 전부 못 넣음
  → 자동 압축(auto-compaction) 적용
  → "작업 기억(Working Memory)"

Layer 3: .ai/current.md (수동 관리)
  → 구조화된 작업 상태: 결정사항, 진행률, 메모
  → "회의록"
```

### Auto-compaction (자동 압축)

컨텍스트 윈도우의 ~80% 초과 시 자동 발동:

```
압축 전: [처음 N개 메시지] + [중간 수십~수백 메시지] + [마지막 N개 메시지]
압축 후: [처음 N개 메시지] + [AI 생성 요약] + [마지막 N개 메시지]
```

- **처음 유지**: 시스템 프롬프트, 초기 지시, 작업 목표
- **마지막 유지**: 가장 최근 작업 맥락
- **중간 요약**: 컨텍스트 윈도우에 다 못 넣으니 AI가 요약

**한계:** 세션이 이어질수록 중간의 손실이 누적됨.

## 기억의 계층 모델

```
영구 기억 (Permanent):  CLAUDE.md, memory/ 파일
  → "이 프로젝트는 이렇게 하기로 했다" (변하지 않는 것)

작업 기억 (Working):    .ai/current.md
  → "지금 이걸 하고 있고, 여기까지 했다" (작업 단위)

일화 기억 (Episodic):   JSONL 세션, episodic-memory
  → "그때 이런 대화를 했었다" (검색 가능한 과거)

순간 기억 (Ephemeral):  컨텍스트 윈도우
  → "방금 이걸 읽었다" (세션 내에서만)
```

**가장 취약한 고리:** 작업 기억 → 영구 기억 전환. 작업이 끝나면 current.md가 리셋되는데, 결정사항/패턴이 CLAUDE.md나 memory/에 반영 안 되면 소실.

## 현재 시스템 분석 (session-manager)

### 강점
- 압축에 의존하지 않음. 사람이 판단해서 중요한 것만 구조화
- 컨텍스트 윈도우 제한과 무관하게 핵심 정보 보존

### 약점
- 수동. 저장 잊으면 날아감
- 어떤 세션에서 했는지 추적 불가 (session_id 미기록)

## 강화 방향

### 방향 1: session_id 연결 (가장 실용적)

current.md에 세션 이력 추가:

```markdown
## 세션 이력
| 날짜 | session_id | 요약 |
|------|-----------|------|
| 03-01 | 0f336571-... | Step 4~5 훅, 서브에이전트 |
| 03-02 | a14d5ff8-... | Step 6 세션 실습 |
```

"무엇을 했는지"(current.md) + "어떻게 했는지"(JSONL 추적) 연결.

### 방향 2: Stop 훅으로 자동 저장

세션 종료 시 자동으로 current.md에 세션 요약 추가.

**문제:** 가벼운 세션(MCP 확인, 플러그인 테스트 등)도 저장되면 노이즈.

**해결: 임계값 기반 필터링**

실제 세션 데이터 측정 결과:

```
세션 A: 154턴, 70파일변경 → 대규모 구현        → 저장 ✅
세션 B:  46턴, 10파일변경 → 학습 + 실습        → 저장 ✅
세션 C:   5턴,  4파일변경 → 키보드 질문         → 스킵 ❌
```

```
턴 <10 & 파일변경 <5:  스킵 (가벼운 세션)
턴 10~20:              물어보기 (borderline)
턴 >20:                자동 저장
```

**단, 엣지 케이스 존재:**
- 파일 변경 0이지만 중요한 설계 논의 (30분 토론)
- 5턴이지만 핵심 버그 원인 발견

→ 완전 자동화보다는 **"적절한 기본값 + /save로 수동 보정"** 이 현실적.

### 방향 3: episodic-memory + JSONL 인덱싱

JSONL 세션 데이터를 인덱싱해서 "그때 뭐 했더라?" 검색 가능하게. 구현 복잡도 높음.

## 결론

- Auto-compaction은 **순간 기억** 레이어의 해결책
- session-manager는 **작업 기억** 레이어의 해결책
- 서로 다른 문제를 풀고 있어서 **둘 다 필요**
- 가장 실용적인 강화: **session_id 기록 + 임계값 기반 자동 저장 + /save 수동 보정**
