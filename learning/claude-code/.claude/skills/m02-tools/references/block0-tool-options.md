# Block 0: 도구의 3가지 옵션

## Phase A — 개념

### 핵심 개념

M01에서 `tools: []`로 도구를 비활성화했다. 도구 없는 에이전트는 순수 LLM이다.
도구를 주면 에이전트는 **스스로 판단해서** 어떤 도구를 쓸지 결정한다.

```
tools: []                        → 순수 LLM (대화만 가능)
tools: ["Read", "Glob", "Grep"]  → 코드베이스 탐색 에이전트
tools: ["Read", "Edit", "Bash"]  → 코드 수정 에이전트
```

### SDK의 도구 관련 옵션 3가지

| 옵션 | 역할 | 비유 |
|------|------|------|
| `tools` | 사용 **가능한** 도구 목록 | "이 도구만 쓸 수 있어" |
| `allowedTools` | 자동 승인 목록 | "이 도구는 물어보지 않고 써도 돼" |
| `disallowedTools` | 명시적 금지 | "이 도구는 절대 쓰지 마" |

### 비유로 이해하기

```
tools         = 도구 상자에 뭘 넣을지 (도구의 존재)
allowedTools  = 허가증 (사용자 승인 없이 자동 실행)
disallowedTools = 잠금 장치 (절대 사용 불가)
```

플러그인에서는 Claude Code가 도구를 자동 제공했다. SDK에서는 `tools`로 **정확히 어떤 능력을 줄지** 직접 제어한다. 이것이 "하네스를 직접 구축"하는 것의 의미.

### 관찰 과제

다음 질문에 답해보자 (코드 실행 전):
- `tools`를 생략하면 어떻게 될까?
- `tools: ["Read"]`와 `tools: ["Read", "Glob"]`의 차이는?

> ⛔ **STOP** — 위 질문에 대한 생각을 공유해주세요.

## Phase B — 피드백 + 퀴즈

### 피드백 포인트

- `tools` 생략 시: 모든 빌트인 도구가 활성화됨 (Read, Edit, Bash, Glob, Grep, Write 등)
- `tools: ["Read"]`: 파일을 읽을 수만 있고, 검색은 불가
- `tools: ["Read", "Glob"]`: 파일 검색 + 읽기 가능. 에이전트가 먼저 Glob으로 파일을 찾고, Read로 읽는 패턴을 스스로 선택

### 퀴즈

**Q1**: `tools: ["Read"]`만 주고 "package.json을 읽어줘"라고 하면 에이전트가 성공할까?

**Q2**: `tools: ["Glob"]`만 주고 "package.json의 내용을 알려줘"라고 하면?

### 정답 가이드

- Q1: 성공한다. 파일 경로를 알고 있으면 Read만으로 충분
- Q2: Glob으로 파일 존재는 확인하지만 내용을 읽을 수 없다. 에이전트가 "Read 도구가 없어서 내용을 확인할 수 없습니다"라고 답할 것
