# Block 0: 서브에이전트 개념

## Phase A — 개념

### 핵심 개념

**Subagent = Claude의 분신.** 독립된 컨텍스트에서 작업을 수행한다.

```
메인 에이전트
  ├── Sub-agent 1: "파일1 리뷰" (독립 컨텍스트)
  ├── Sub-agent 2: "파일2 리뷰" (독립 컨텍스트)
  └── Sub-agent 3: "파일3 리뷰" (독립 컨텍스트)
```

### 서브에이전트의 장점

| 장점 | 설명 |
|------|------|
| 전문화 | 특정 작업에 특화된 에이전트 생성 |
| 컨텍스트 분리 | 각 서브에이전트가 독립된 컨텍스트 → 메인 대화는 깔끔 |
| 역할 분담 | 메인이 조율, 서브가 실행 |

### SDK `agents` 옵션 구조

```typescript
agents: {
  "agent-name": {
    description: string,  // 자연어 설명 (메인 에이전트가 읽고 판단)
    prompt: string,       // System Prompt (서브에이전트의 역할)
    tools?: string[],     // 사용 가능한 도구 (생략 시 부모 상속)
    model?: string,       // 모델 (생략 시 부모 상속)
    maxTurns?: number,    // 턴 수 제한
  }
}
```

### 플러그인 agents/*.md와의 관계

```
플러그인: agents/reviewer.md  → 선언적 (description + 본문)
SDK:     agents: { reviewer: { description, prompt, tools } }  → 프로그래밍적
```

메인 에이전트는 서브에이전트의 `description`을 읽고, 작업에 적합한 서브에이전트를 **스스로 선택**한다.

### 사전 질문

- 서브에이전트에게 메인과 다른 도구를 줄 수 있을까?
- 서브에이전트가 또 다른 서브에이전트를 호출할 수 있을까?

> ⛔ **STOP** — 위 질문에 대한 생각을 공유해주세요.

## Phase B — 피드백 + 퀴즈

### 피드백 포인트

- 다른 도구: 가능하다. `tools` 필드로 서브에이전트별 도구를 독립 설정
- 중첩 호출: 가능하지만 권장하지 않음. 복잡도 폭증 + 비용 증가

### 퀴즈

**Q1**: 서브에이전트의 `description`은 누가 읽는가? 사용자? 메인 에이전트?

### 정답 가이드

- Q1: **메인 에이전트**가 읽는다. description을 보고 "이 작업에 이 서브에이전트가 적합하다"고 판단
