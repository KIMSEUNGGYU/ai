# 코드/도구 컨벤션

> 항상 적용되는 맥락 무관 규칙. 자동 관리 — /done 시 트리거 D(반복 교정)로 추출.

## 확인 후 응답 원칙 (최우선)
확인하지 않은 것을 확인한 것처럼 말하지 않는다. 속도보다 정확성 우선.

체크리스트:
- 파일 내용 언급 시 → 실제로 읽었는가?
- "~일 것입니다", "아마" 표현 금지 → "확인 결과 ~입니다" 또는 "확인이 필요합니다"
- 불확실한 부분이 있으면 → 답변 전에 확인 먼저
- "완료", "문제없다" 선언 전 → 관련 파일 전체 확인 + 호출 체인 끝까지 추적
- 파일 변경 후 → 다시 읽어서 실제 상태 확인
- 모르면 → "모릅니다" / "확인이 필요합니다"라고 명확히 밝힘

근거: 빠르게 답하려다 검증 없이 그럴듯한 답을 만들어내는 패턴이 반복됨.
"아 사실은..." 패턴은 신뢰를 깨뜨림. (2026-03-14, 통합 2026-04-03)

## 작업 유형별 필수 도구
아이디어/설계/구조 설계 시:
- `brainstorming` 스킬 (superpowers) — 아이디어 → 설계 → 스펙 문서화 프로세스
- `writing-plans` 스킬 (superpowers) — 구현 계획 작성

플러그인/스킬/커맨드/훅 생성 및 수정 시:
- `plugin-dev:*` 스킬 — 플러그인/스킬/훅/커맨드/에이전트 개발 가이드
- `skill-creator` 스킬 — 스킬 생성, 개선, 평가(eval), description 최적화
- `claude-code-guide` 에이전트 — Claude Code 공식 문서 기반 구조/패턴 확인

임의로 추측하지 말고 해당 도구를 먼저 사용한 뒤 작업. (2026-03-21, 2026-04-03 확장)

## 결정 시 근거 필수
어떤 결정이든 반드시 근거를 함께 제시한다. 근거 없이 결론만 말하지 않는다. (2026-03-21)

## 기존 구조 제거부터 검토
기존 파일/구조를 수정할 때 "어떻게 줄일까"가 아니라 "이게 필요한가?"를 먼저 판단한다.
"유지하면서 개선"보다 "제거"를 먼저 검토. 관리 포인트 최소화 원칙. (2026-03-21)

## 표준/공식 API 최우선
라이브러리/SDK의 공식 타입에 없는 옵션을 타입 단언이나 우회로 사용하지 않는다.
공식 지원이 없으면 해당 기능을 사용하지 않고, 지원될 때까지 기존 방식을 유지한다. (2026-04-03)

## 라이브러리 내장 API 우선
직접 구현 전 사용 중인 라이브러리의 내장 API를 먼저 검토.
예: `form.setValue`로 초기값 세팅 대신 `form.resetField` 사용. (ISH-1313)

## hq 볼트 문서 frontmatter 필수
hq(~/hq/)에 문서를 생성하거나 이동할 때 반드시 CLAUDE.md의 frontmatter 규칙에 따라 `tags`, `date`, `status`를 포함한다.
기존 문서에 frontmatter가 없으면 추가한다. (2026-03-18)
