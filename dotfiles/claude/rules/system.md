# 시스템 운영 규칙

> AI 행동 규칙, git 워크플로우, 자가학습 메타. 자동 관리 — /done 시 트리거 B(이유 설명)로 추출.

## 자가학습

### 실행 원칙
- 자가학습은 session_id 유무와 관계없이 항상 수행. session_id 없으면 현재 세션 transcript 사용.
- active 파일 0개여도 자가학습은 정상 실행. "종료"로 스킵하지 않음.
- 판단이 필요한 작업 시작 시 rules 파일을 먼저 읽고 확인 후 행동.
- recap/자가학습 시 transcript를 일부만 읽고 건너뛰지 않는다. 전체를 다 읽어야 판단 기록 품질이 보장됨. (2026-03-25)
- transcript 전처리에서 스킬 프롬프트처럼 매번 동일하게 반복 주입되는 콘텐츠는 한 줄 요약으로 치환한다. 노이즈 30%+ 시 분석 품질 저하. (2026-03-25)

### 학습 대상 파일과 트리거
- `decisions.md` — 트리거 A(거부), B(이유 설명), C(선택지 결정)
- `conventions.md` — 트리거 D(반복 교정)
- `system.md` — 트리거 B(이유 설명, 시스템 관련)
- `identity.md` — 반자동 (AI 제안 → 사용자 승인)

### 분류 기준
- decisions: 상황에 따라 달라지는 판단 (맥락 의존적)
- conventions: 항상 적용되는 규칙 (맥락 무관)
- system: 시스템/도구 운영 방법
- 메모리 vs rules 파일: 반복 적용 규칙 → rules 파일(conventions.md/decisions.md), 프로젝트/세션 한정 맥락 → 메모리
- convention으로 해결 가능하면 메모리 사용 금지. 관리 포인트 최소화. (2026-03-15)

### rules 파일 형식
모든 rules 파일은 자연어(제목 + 설명 2~3줄)로 통일. WHEN/DO/WHY 같은 구조화된 형식 사용 금지.
자연어가 AI 파싱에 더 효율적이고, 모든 케이스에 맞는 단일 형식이 없기 때문. (2026-03-15)
내용은 코드 패턴(어떻게)보다 판단 근거(왜)를 중심으로 기록. 근거를 알면 새로운 상황에서도 판단 가능. (2026-03-19)

### 감독 방식
- 자동 저장 (매번 확인 안 함)
- 기존 패턴과 모순 시 → 사용자 확인
- 주기적 감독 (주 1회 또는 사용자 요청 시) — 불필요한 것 삭제, 모순/중복 정리

## 진행 스타일
- 완료 확인 질문보다는 결과를 간결하게 보여주고 다음 단계로 넘어간다.
- 큰 작업은 한 번에 하지 말고 점진적·단계별로 진행.
- 탐색 요청('찾아봐') 시 확인 질문 없이 바로 실행.
- 판단이 수반되는 모든 행위(저장, 이동, 삭제, 분류) 시 반드시 왜 그 행위를 했는지 근거를 함께 설명.

## 문서 동기화
- PC 구조, 설정, 워크플로우 변경 시 관련 문서도 함께 업데이트.

## specs 관리
- 완료된 작업의 spec 파일은 삭제. CHANGELOG 요약으로 충분, specs는 진행 중 작업용.

## 브랜치/환경
- worktree 사용 시 env 복붙 등 추가 설정이 번거로우므로, 간단한 수정은 stash + 브랜치 전환 선호.

## 파일 구조 변경
- 파일 구조 변경 시 소유권(어떤 플러그인/시스템이 관리하는지)과 스코프(전역/프로젝트)를 먼저 확인한 뒤 제안한다.

## 경로 규칙
- 파일 저장 위치 결정 시 pc-map.md 규칙을 반드시 먼저 참조.

## 플러그인 버전
- 플러그인 변경 후 반영 시점: exit 후 새 세션에서 반영됨을 명시.
- plugin.json에 별도 date 필드를 추가하지 않는다. git log가 이미 시점 기록 (SSOT). (2026-03-15)

## Git 워크플로우

### 커밋 메시지
```
<type>: <description>
<optional body>
```
Types: feat, fix, refactor, docs, test, chore, perf, ci
Attribution disabled globally via ~/.claude/settings.json.

### PR 워크플로우
1. PR 생성 전 반드시 `/simplify` 실행
2. 전체 커밋 히스토리 분석 (최신 커밋만이 아닌 전체)
3. `git diff [base-branch]...HEAD`로 모든 변경 확인
4. 종합 PR 요약 작성
5. 테스트 계획 포함
6. 새 브랜치면 `-u` 플래그로 push
