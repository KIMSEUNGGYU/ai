# CC 사용 전체 생명주기 + cc-monitor 수집 지점

## 전체 흐름

```
[1] 세션 시작 (claude 실행)
    │
    │  CC 내부: 세션 생성, CLAUDE.md 로드, 플러그인 로드
    │
    ├→ Hook: SessionStart
    │  stdin: { session_id, cwd, model }
    │  send-event.ts 실행:
    │    - collectConfig(cwd) → CLAUDE.md/rules/MCP/hooks/active_task 스캔
    │    - API 전송 → Session 테이블 생성
    │  ★ 수집: 세션 환경 스냅샷
    │
    ▼
[2] 사용자 프롬프트 입력
    │
    ├→ Hook: UserPromptSubmit (settings.json)
    │  stdin: { prompt, session_id }
    │  send-event.ts 실행:
    │    - prompt 앞 500자 + 마스킹 → API 전송
    │  ★ 수집: 프롬프트 텍스트
    │
    ├→ Hook: UserPromptSubmit (fe-workflow hooks.json, matcher 매칭 시)
    │  stdin: { prompt, session_id, cwd }
    │  fe-convention-prompt.sh 실행:
    │    - prompt에서 키워드 grep
    │    - 매칭 시: cat convention파일 → stdout (CC가 context에 주입)
    │    -         curl로 메타데이터 → cc-monitor API
    │    - 미매칭: 5줄 요약만 stdout
    │  ★ 수집: 어떤 컨벤션이 주입됐는지 (파일명, 바이트, 키워드)
    │  ⚠ 미수집: stdout 전문 (CC context에 들어간 실제 내용)
    │
    │  CC 내부 (블랙박스):
    │    - hook stdout → system-reminder로 context에 적재
    │    - CLAUDE.md, rules, 대화 히스토리 등과 함께 API 호출
    │
    ▼
[3] AI 응답 생성 (도구 호출 루프)
    │
    │  AI가 도구를 호출할 때마다:
    │
    ├→ Hook: PreToolUse (settings.json)
    │  stdin: { tool_name, tool_input, tool_use_id }
    │  send-event.ts 실행:
    │    - 경량 도구 (Read/Edit/Glob/Grep/Write) → 무시
    │    - 비경량 도구 → API 전송
    │  ★ 수집: 도구 호출 시작 시점 (duration 계산용)
    │
    │  CC 내부: 도구 실행 (파일 읽기/수정, Bash, Agent 등)
    │
    ├→ Hook: PostToolUse (settings.json)
    │  stdin: { tool_name, tool_input, tool_use_id }
    │  send-event.ts 실행:
    │    - 경량 도구 → /tmp/cc-monitor-{sessionId}.json에 카운트 +1
    │    - 비경량 도구 → API 전송 + Pre와 매칭해서 duration 계산
    │  ★ 수집: 도구 사용 완료 + 실행 시간
    │
    ├→ Hook: PostToolUse (fe-workflow hooks.json, Edit|Write 시)
    │  post-edit-convention.sh 실행:
    │    - 파일 확장자로 convention 분류 (.tsx→coding-style 등)
    │    - stderr: 체크 리마인더 → AI에게 전달
    │    - curl: 메타데이터 → cc-monitor API
    │  ★ 수집: 어떤 파일 수정 후 어떤 체크 발동
    │
    ├→ Hook: PostToolUse (fe-workflow hooks.json, Edit|Write 시)
    │  harness-check.sh 실행:
    │    - 9개 규칙 기계적 검증 (파일 경로, 코드 패턴)
    │    - 위반 시 stderr → AI가 자동 수정
    │  ⚠ 미수집: 위반 결과가 cc-monitor로 안 감
    │
    │  ... (도구 호출 반복) ...
    │
    ▼
[4] AI 응답 완료 (턴 종료)
    │
    ├→ Hook: Stop (settings.json)
    │  stdin: { transcript_path }
    │  send-event.ts 실행:
    │    - /tmp 카운터 파일 읽기 → tool_summary 생성
    │    - transcript JSONL 파싱 → 토큰 사용량 합산
    │    - API 전송 후 /tmp 카운터 삭제
    │  ★ 수집: 도구 사용 요약 (경량 도구 포함) + 토큰 사용량
    │
    ▼
[5] 다음 프롬프트 → [2]로 반복
    │
    ▼
[6] 세션 종료 (/exit, 터미널 닫기)
    │
    ├→ Hook: SessionEnd (settings.json)
    │  send-event.ts 실행:
    │    - Stop과 동일 + ended_at, status: "ended" 설정
    │  ★ 수집: 세션 종료 시각 + 최종 토큰/도구 요약
    │
    ▼
[끝]
```

## 대시보드 탭 ↔ 수집 지점 매핑

| 탭 | 데이터 소스 |
|------|------------|
| Overview | [4]의 토큰/비용 + [3]의 도구 통계 |
| Sessions | [1]의 세션 목록 + [6]의 종료 정보 |
| History | [2]의 프롬프트 기준으로 [3][4]를 그룹핑 |
| Cost | [4]의 토큰 → 달러 환산 |
| Config | [1]의 config_snapshot |

## 블랙박스 영역 (수집 불가)

| 영역 | 설명 |
|------|------|
| hook stdout → context 주입 과정 | CC가 stdout을 실제로 context에 넣는 내부 로직 |
| CLAUDE.md/rules 적재 방식 | context에 어떤 순서/형태로 들어가는지 |
| 모델 해석 | context에 있는 컨벤션을 모델이 어떻게 해석/적용하는지 |
| compaction | 압축 시 뭐가 남고 뭐가 날아가는지 |
| 서브에이전트 내부 | Agent 도구로 위임한 작업의 내부 동작 |

## 수집 소스 요약

|------|----------|------|
| settings.json hooks | send-event.ts | ~/dev/ai/services/cc-monitor/collector/ |
