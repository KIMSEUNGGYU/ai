# cc-monitor 데이터 수집 아키텍처

## 데이터 소스

cc-monitor가 수집하는 데이터 소스는 2개.

### 소스 1: settings.json hooks → send-event.ts

`~/.claude/settings.json`에 등록된 hook이 Claude Code 이벤트마다 `send-event.ts` 실행.
경로: `~/dev/ai/services/cc-monitor/collector/send-event.ts`

### 소스 2: fe-workflow plugin hooks → 직접 curl

fe-workflow 플러그인의 bash 스크립트가 cc-monitor API로 직접 전송.
경로: `~/dev/ai/plugins/fe-workflow/hooks/scripts/`

## 소스 1 상세: settings.json hooks

### SessionStart
- **언제**: 세션 시작, resume, /clear, compact 시
- **어떻게**: Claude Code가 `{ session_id, cwd, hook_event_name: "SessionStart", model }` 전달 → `send-event.ts`가 `collectConfig(cwd)` 실행해서 config_snapshot 첨부 → API 전송
- **config_snapshot 수집 방법**:
  - CLAUDE.md: 5개 후보 경로 존재 확인 (`~/.claude/CLAUDE.md`, `{cwd}/CLAUDE.md` 등)
  - rules: `~/.claude/rules/`, `{cwd}/.claude/rules/` 디렉토리 스캔
  - MCP: `settings.json`, `.mcp.json` 등에서 `mcpServers` 키 파싱
  - hooks: `settings.json`에서 `hooks` 키 파싱
  - active_task: `{cwd}/.ai/active/*.md` 중 최신 수정 파일명
- **목적**: 세션 생성 + 설정 환경 추적 + task_name 자동 분류

### PreToolUse
- **언제**: AI가 도구 호출 직전
- **어떻게**: `{ tool_name, tool_input, tool_use_id }` 전달 → 경량 도구(Read/Edit/Glob/Grep/Write)면 무시. 비경량 도구만 API 전송
- **목적**: Pre→Post duration 계산용 (활용도 낮음, 경량화 대상)

### PostToolUse
- **언제**: AI가 도구 호출 완료 후
- **어떻게**: `{ tool_name, tool_input, tool_use_id }` 전달 → 경량 도구면 `/tmp/cc-monitor-{sessionId}.json`에 카운트 +1만 하고 API 전송 안 함. 비경량 도구는 API 전송 + Pre의 tool_use_id와 매칭해서 duration 계산
- **목적**: 도구 사용 추적 (경량 도구는 Stop 시 일괄 전송)

### UserPromptSubmit
- **언제**: 사용자가 프롬프트 제출할 때
- **어떻게**: `{ prompt }` 전달 → `send-event.ts`가 prompt 앞 500자 추출 + 민감정보 마스킹(API키, 이메일 등) → API 전송
- **목적**: History 탭의 턴 기준점 + 프롬프트 기록

### Stop
- **언제**: AI 응답 완료 시 (턴 종료)
- **어떻게**: `{ transcript_path }` 전달 → `send-event.ts`가:
  1. `/tmp/cc-monitor-{sessionId}.json` 읽어서 tool_summary(경량 도구 카운트) 첨부
  2. transcript_path JSONL 파싱해서 토큰 사용량 합산 → transcript_usage 첨부
  3. API 전송 후 임시 파일 삭제
- **목적**: 도구 사용 요약 + 토큰/비용 계산

### SessionEnd
- **언제**: 세션 종료 시 (`/exit`, 터미널 닫기 등)
- **어떻게**: Stop과 동일한 처리 + `ended_at`, `status: "ended"` 설정
- **목적**: 세션 라이프사이클 종료

## 소스 2 상세: fe-workflow plugin hooks

### fe-convention-prompt.sh
- **언제**: UserPromptSubmit 이벤트에서 hooks.json matcher 매칭 시
- **어떻게**:
  1. Claude Code가 `{ prompt, session_id, cwd }` stdin으로 전달
  2. prompt에서 키워드 grep → 매칭되면 해당 convention 파일 `cat`으로 stdout 출력 (→ Claude context 주입)
  3. 동시에 백그라운드로 curl: `{ plugin_name, hook_name, injected_conventions, injection_bytes, matched_keywords }` → cc-monitor API
  4. 매칭 안 되면 5줄 요약만 stdout 출력, curl 안 함
- **목적**: "어떤 컨벤션이 주입됐는지" 추적 (Plugin Health, History 탭)

### post-edit-convention.sh
- **언제**: PostToolUse 이벤트에서 tool_name이 Edit|Write일 때
- **어떻게**:
  1. Claude Code가 `{ tool_name, tool_input, session_id, cwd }` 전달
  2. `tool_input.file_path` 확인 → .tsx면 coding-style, *remote*/*dto*면 api-layer 등 분류
  3. 분류된 타입의 체크 메시지를 stderr로 출력 (→ AI에게 리마인더)
  4. 백그라운드로 curl: `{ plugin_name, hook_name, injected_conventions, injection_bytes, target_file }` → cc-monitor API
- **목적**: "파일 수정 후 어떤 체크가 발동됐는지" 추적

## 수집 안 하는 것

| 데이터 | 이유 |
|--------|------|
| hook stdout (컨벤션 전문) | 매번 ~3000토큰을 DB에 저장하면 낭비 |
| transcript 전문 | 용량 너무 큼. 토큰 수치만 추출 |
| tool_output (도구 실행 결과) | Claude Code가 hook에 전달 안 함 |
| harness-check 위반 결과 | 아직 미구현 (stderr만 출력 중) |

## 흐름도

```
사용자 프롬프트 입력
  ├→ [settings.json] UserPromptSubmit → send-event.ts → API (prompt_text)
  └→ [hooks.json] matcher 매칭 → fe-convention-prompt.sh
       ├→ stdout: 컨벤션 전문 (→ Claude context)
       └→ curl: 메타데이터 (→ cc-monitor API)

AI 도구 호출
  ├→ [settings.json] PreToolUse → send-event.ts → API or 스킵
  └→ [settings.json] PostToolUse → send-event.ts → API or 로컬 카운팅
       └→ [hooks.json] Edit|Write → post-edit-convention.sh
            ├→ stderr: 체크 리마인더 (→ AI)
            └→ curl: 메타데이터 (→ cc-monitor API)

AI 응답 완료
  └→ [settings.json] Stop → send-event.ts
       ├→ /tmp 카운터 읽기 → tool_summary
       └→ transcript 파싱 → transcript_usage → API
```

## 키워드 매칭 설정

### hooks.json matcher (hook 발동 조건)
```
remote|mutation|query|API|api|dto|DTO|schema|컴포넌트|페이지|구현|추가해|작성|
연동|통합|리팩토링|refactor|리뷰|review|hook|훅|useEffect|폴더|folder|error|
에러|fetch|endpoint|form|zod|추상화|분리|추출|개선|정리|구조|component|handler
```

### fe-convention-prompt.sh 내부 매칭 (어떤 컨벤션을 주입할지)

| 키워드 | 주입 파일 |
|--------|-----------|
| remote, mutation, query, API, dto, fetch, httpClient, endpoint 등 | api-layer.md |
| component, 컴포넌트, useEffect, useState, hook, form, zod 등 | coding-style.md |
| 리팩토링, refactor, 리뷰, review, 추상화, 분리, 개선 등 | code-principles.md |
| 폴더, folder, 구조, structure, 파일 생성, models/, remotes/ 등 | folder-structure.md |
| error, 에러, 예외, ErrorBoundary, try catch, Suspense 등 | error-handling.md |

최대 2개까지 주입. 매칭 안 되면 5줄 요약만 출력.
