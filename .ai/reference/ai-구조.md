Claude Code로 작업하다 세션이 날아간 적이 있습니다. 컨텍스트 초과로 대화가 끊겼는데, 직전까지 뭘 시켰고 어디까지 진행됐는지 정확히 복원할 수가 없었습니다. 그때 생각했습니다 — 내가 AI에게 시킨 일이 어딘가에 쌓이고 있어야 하는 거 아닌가.

—

비개발자로서 한 가지 더 불안한 게 있었습니다. 내가 혹시 이상한 방식으로 하고 있어도, 그걸 짚어줄 사람이 없다는 건데요. "근본없는 방식이 누적되면 어쩌나" 걱정되기도 했구요. 여러 작업들을 동시에 돌려보고 있는데, 곳곳에 중복해서 불필요한 게 끼어 있는 건 아닌지 — 검토하고 싶었는데 검토할 수단이 없었습니다.

—

연초에 만난 호진오라버님이 audit 파일을 만들어 관리하시는 걸 봤는데 "한번 해봐야겠다"생각만 하고 미루고 있었는데요. 새로 또 풀어야 할 문제가 생겼기에 이참에 만들어봐야겠다 다짐하게 되었습니다. (다른거 하나 돌 때 세션이 오래걸리니까, 남는 시간이 아까워서 기록을 하든 논문을 읽든 뭐라도 할 일을 채워놓게 되네요)

프롬프트를 기록하는 것, 거기에 AI가 실제로 뭘 했는지 작업 로그를 붙이는 것, 이 둘을 시간순으로 연결하면 꽤 쓸모 있는 게 나오지 않을까 싶었습니다.

—

그래서 Claude Code의 훅(hook) 시스템을 이용해서 만들어봤습니다. 4개의 훅이 하나의 로그 파일에 기록합니다.

- session-start.py → 세션 시작 시각, 브랜치, 프로젝트 경로
- prompt-audit.py → 내가 입력한 프롬프트를 통째로 기록
- worklog.py → AI가 실행한 모든 도구를 한 줄씩 기록
- session-end.py → 세션 종료 시각, 종료 사유

도구별로 기록하는 내용이 다릅니다. 파일 도구(Write/Edit/Read)는 경로를, Bash는 실행 커맨드를, Grep은 검색 패턴을, Task는 에이전트 설명을 남깁니다. 사실 화면에 매일 Bash, Grep 떴는데 그냥 특수문자처럼 넘겼지 의미도 이번에 알았네요. 

이게 합쳐지면 이런 로그가 됩니다.

━━━ SESSION START 14:30 [main] project ━━━
:memo: PROMPT: "ops-guide.html에 폴더 구조 추가해줘"
 14:30:12 [Read] ops-guide.html
 14:30:15 [Grep] "matrix-table" in ops-guide.html
 14:30:18 [Edit] ops-guide.html
:memo: PROMPT: "직접 네틀리파이에 배포해줘"
 14:31:02 [Skill] netlify-deploy
 14:31:15 [Bash] npx netlify deploy --prod
━━━ SESSION END 14:35 ━━━

프롬프트 → 작업 → 결과가 시간순으로 연결됩니다.

—

여기서 한 단계 더 갔습니다. 세션이 끝날 때 자동으로 분석 리포트를 만들어주면 좋겠다 싶어서, 기존에 잘 쓰고 있던 구봉님의 /wrap 스킬에 worklog-reviewer 에이전트를 추가했습니다. 기존 4개 병렬 에이전트에 5번째가 붙은 겁니다. 

이 에이전트가 하는 일은 단순합니다. 워크로그에서 현재 세션을 파싱해서 통계를 뽑아줍니다.
- 프롬프트 몇 개 던졌는지
- 어떤 도구를 몇 번 썼는지 (Edit 20회, Bash 8회...)
- 가장 많이 건드린 파일이 뭔지 (파일 핫스팟)
- 세션 지속 시간

리포트는 ~/claude-session-reports.log에 누적됩니다.

—

그리고 하나 더 — 도구 사용 패턴에서 스킬을 추천하는 기능도 붙였습니다. 이전에 추천받아 사용하고 있던 /find-skills를 결합하였습니다. 

.py 파일을 3번 이상 편집했으면 /python-review와 /tdd를, Bash 비율이 30% 이상이면 /systematic-debugging을, .html 편집이 2회 이상이면 /web-design-guidelines를 추천합니다.

작업 패턴이 곧 "지금 이 세션에서 뭘 하고 있는지"를 말해주거든요. 그걸 기반으로 "이 스킬 써보세요"를 자동으로 제안하는 겁니다.

—

:wrench: 실제로 돌려보니 몇 가지가 보이더라고요.

같은 프롬프트가 3~4번 반복되는 구간이 있었습니다. 돌아보니 그 지점이 정확히 "뭔가 안 풀려서 삽질하던 구간"이었어요. 프롬프트 로그를 보지 않았으면 그냥 "오래 걸렸다" 정도로만 기억했을 겁니다.

프로젝트별로 도구 분포가 확연히 다르다는 것도 보였습니다. 어떤 프로젝트에서는 Read+Grep이 70%인데 (탐색 위주), 다른 프로젝트에서는 Edit이 60%입니다 (구현 위주). 이게 프로젝트 성격을 숫자로 보여주더라고요.

제일 쓸모 있었던 건 "이 작업 왜 했지?" 추적입니다. 세션이 날아가거나, 며칠 뒤에 돌아왔을 때, 로그를 읽으면 맥락이 바로 복원됩니다. 이건 진짜 만들기 전에는 몰랐던 효용이었어요.

—

성능 영향은 거의 없습니다. 각 훅이 하는 일은 파일에 한 줄 append하는 게 전부라서, 체감되는 지연은 없었습니다. 다른 터미널에서 tail -f 명령어를 걸어두면 실시간으로 AI가 뭘 하고 있는지 볼 수 있는 건 덤이고요.

—

결국 이건 "AI와 일하는 나를 관찰하는 도구"였습니다. 만들기 전에는 "기록이 쌓이면 좋겠다" 정도였는데, 실제로 며칠 쌓이니까 보이는 게 달라졌습니다. 내가 어떤 프로젝트에서 어떤 식으로 AI를 쓰고 있는지, 어디서 막히는지, 어디서 뻘짓을 하는지. AI를 잘 쓰는 것보다 내가 어떻게 쓰고 있는지 아는 게 먼저라는 걸 — 로그가 알려줬습니다. 

이제 준비는 되었으니, 다시 또 로그를 쌓으러 가봐야겠네요... :melting_face:

Github 역대 최고 속도 스타 달성한 개발자가 공개한 "AI 코딩 10가지 원칙"

OpenClaw(ex. Clawdbot) 창시자 Peter Steinberger 의 인터뷰를 보고 적잖이 충격 받았어요. 

그는 13년간 60~70명 규모 회사를 운영하고 매각한 후, 3년 휴식 끝에 복귀한 베테랑인데요. 그가 말하는 AI 시대 개발은 우리가 아는 것과 완전히 달랐습니다.

그의 시작은 단순했어요.

처음부터 거창한 사업 계획이 없었어요. 단순히 "AI 를 가지고 놀아보자" 는 마음으로 시작했고, 외출 중에도 WhatsApp 으로 집 컴퓨터와 대화하고 싶어서 이 도구를 만들었다고 해요.

결정적인 "아하 모멘트" 가 있었어요.

여행 중 에이전트에게 음성 메시지를 보냈는데, 음성 지원 기능을 코딩한 적이 없었거든요. 그런데 에이전트가 스스로 Opus 파일 형식을 파악하고, ffmpeg 를 찾아 변환하고, API 키를 찾아 번역까지 해서 답을 보냈대요. 이 순간 그는 에이전트가 "영리하고 수완 좋은 야수" 임을 깨달았다고 합니다.

이런 경험을 바탕으로 Peter 가 정리한 AI 코딩 10가지 원칙이에요.

:one: 완벽주의를 버려야 AI 와 일할 수 있어요 

• 70명 팀을 이끌며 "내 스타일대로 안 나와도 받아들이는 법"을 배움 
• 코드가 내 취향과 100% 일치하지 않아도 동작하면 넘어가는 유연함이 필요 
• 이 훈련이 지금 에이전트와 협업할 때 가장 큰 자산이 됨

:two: 에이전트가 스스로 검증할 수 있게 설계해요 

• Peter 는 이걸 "루프를 닫는다(Close the loop)" 고 표현 
• 컴파일, 린트, 실행, 검증까지 에이전트가 직접 수행하도록 시스템 구성 
• 사람이 중간에 확인해주는 구조는 병목이 되어 전체 속도를 떨어뜨림

:three: Pull Request 는 죽었고, Prompt Request 시대예요 

• 코드 자체보다 그 코드를 만든 프롬프트가 더 중요해짐 
• 외부 PR 은 거의 거절하고, 핵심 아이디어만 추출해서 프롬프트로 재활용 
• 그의 형제도 같은 방식으로 일한다고 하니, 이미 퍼지고 있는 패턴인 것 같아요

:four: 코드 리뷰 대신 아키텍처 토론만 해요 

• Discord 에서도 핵심 팀원과 코드 이야기는 안 함 
• 대신 시스템 구조, 큰 결정, 방향성에 대해서만 논의 
• 구현 디테일은 에이전트 몫이라는 관점이 팀 전체에 정착됨

:five: 5~10개 에이전트를 동시에 돌려요 

• 하나의 작업에 오래 매달리지 않고 여러 작업을 병렬로 큐잉 
• 계획 세우고 에이전트에게 맡기고, 바로 다음 에이전트로 이동 
• 이렇게 해야 "몰입 상태(Flow state)" 를 유지할 수 있다고 함

:six: 계획 단계에 놀라울 정도로 긴 시간을 써요 

• 에이전트와 충분히 핑퐁하며 계획을 다듬음 
• 도전하고, 수정하고, 반박하고, 만족스러울 때까지 반복 
• Codex 선호 이유: Claude Code 는 실행 중 확인 질문을 해서 흐름이 끊김 
• 계획이 탄탄하면 실행 단계에서 개입할 필요가 거의 없어짐

:seven: 일부러 모호하게 지시해요 

• 너무 구체적으로 지시하면 AI 가 그 범위 안에서만 움직임 
• 의도적으로 여지를 남겨서 자신이 생각 못한 방향을 발견하게 함 
• 제가 직접 해보니 정말 효과 있는 방식이에요. 예상 못한 해결책이 나올 때가 많아요. 물론 항상 이렇게 할 필요는 없습니다

:eight: 원격 CI 10분 기다리지 말고 로컬에서 바로 테스트해요 

• 원격 CI 파이프라인 대기 시간 10분이 아깝다는 관점 
• 에이전트가 직접 로컬에서 테스트를 돌리게 설계 
• 피드백 루프를 최대한 짧게 유지해야 반복 속도가 빨라짐

:nine: 대부분의 코드는 지루한 데이터 변환일 뿐이에요 

• 앱 코드 대부분은 "데이터를 이 형태에서 저 형태로 바꾸는 것" 
• 이런 작업에 집착할 필요 없이 에이전트에게 위임 
• 에너지는 코드 자체가 아니라 시스템 설계에 집중해야 함

:keycap_ten: 제품 출시를 좋아하는 사람이 AI 적응에 유리해요 

• 알고리즘 퍼즐 푸는 걸 좋아하는 개발자가 오히려 AI 전환에 힘들어함 
• 구현 디테일보다 결과물과 출시에 관심 있는 사람이 빨리 적응 
• 이건 저도 주변에서 자주 보는 패턴이에요

Peter 의 미래 전망도 흥미로워요

그는 수많은 앱이 사라지고 API 만 남을 거라고 예측해요. MyFitnessPal 을 열어서 입력할 필요 없이, 에이전트에게 음식 사진만 보내면 알아서 칼로리 계산하고 건강 목표를 수정해주는 시대가 온다는 거예요.

여러 논쟁이 있을 수 있지만 결국 Peter 의 10가지 원칙이 말하는 건 하나예요. 완벽주의를 버리고, 코드 리뷰 대신 아키텍처를 논하고, 에이전트가 스스로 검증하게 만들고, 여러 에이전트를 동시에 돌리는 것. 

이 모든 게 "개발자가 직접 코드를 짜지 않아도 되는 환경을 만든다" 는 방향으로 수렴합니다. 그렇다면 AI 시대 진짜 실력은 코드를 잘 짜는 게 아니라, 코드를 짜지 않고도 문제를 해결하는 판을 설계하는 겁니다.

에이전트가 100% 코드를 쓰고 Repository 세팅 방법이 공개됐습니다. "Code Factory" 패턴인데, 핵심은 사람이 아니라 Repository가 정책을 강제한다는 겁니다. 코딩 에이전트가 코드를 쓰고, 리뷰 에이전트가 검증하고, 증거가 기계적으로 확인되고, 발견된 문제가 반복 가능한 테스트 케이스로 전환되는 루프입니다. 이론이 아닙니다. OpenAI Codex 팀은 이 접근법으로 5개월 만에 수동 코드 0줄, 100만 줄의 프로덕트를 만들었고, 엔지니어 1인당 하루 3.5 PR을 처리하고 있습니다.

1. 하나의 JSON 계약서로 모든 정책을 통합

경로별로 리스크 티어를 나눕니다. 예를 들어 app/api/legal-chat/ 같은 핵심 경로는 high tier, 나머지는 low tier. 각 티어별로 필수 체크를 정의합니다. high tier면 risk-policy-gate, harness-smoke, Browser Evidence, CI Pipeline 네 가지를 모두 통과해야 머지됩니다.

이 계약서가 해결하는 건 "silent drift"입니다. 스크립트는 A를 체크하는데 워크플로우는 B를 체크하고 정책 문서는 C를 말하는, 세 곳이 따로 노는 상황. JSON 하나에 넣으면 이게 구조적으로 불가능해집니다. Codex 팀도 같은 교훈을 얻었는데, 에이전트가 볼 수 없는 정보는 존재하지 않는 것과 같다는 겁니다. 슬랙 논의, 구글 독스, 사람 머릿속 맥락은 에이전트에게 접근 불가합니다. 모든 정책과 맥락은 리포 안에 versioned artifact로 있어야 합니다.

2. Preflight gate로 CI 비용을 절감

비싼 test/build/security 작업을 돌리기 전에, risk-policy-gate를 먼저 실행합니다. 정책 위반이 있거나 리뷰가 미해결이면 CI fanout 자체를 시작하지 않습니다. 이미 실패가 확정된 PR에 CI 비용을 쓸 이유가 없으니까요.

3. Current-head SHA discipline (가장 큰 실전 교훈)

Carson이 "가장 큰 practical lesson"이라고 직접 말한 부분입니다. 리뷰 결과는 현재 PR head commit에 매칭될 때만 유효합니다. 오래된 SHA에서 나온 "clean" 결과를 믿고 머지하는 게 실전에서 가장 위험한 함정입니다.

매 push마다 리뷰를 재실행하고, 오래된 코멘트는 무시하고, 최신 리뷰가 실패하면 머지를 차단합니다. 이 원칙을 타협하면 안 됩니다.

4. Remediation agent로 자동 수정

리뷰에서 actionable한 피드백이 나오면, 코딩 에이전트가 자동으로 리뷰를 읽고, 코드를 패치하고, 로컬에서 검증하고, fix commit을 같은 브랜치에 push합니다. 그러면 PR synchronize가 다시 정상 리뷰 경로를 탈 수 있습니다.

여기서 핵심 가드레일이 세 가지입니다. 모델과 effort를 고정해서 재현성을 보장할 것. 현재 head에 매칭되지 않는 stale 코멘트는 무시할 것. 정책 게이트를 절대 우회하지 않을 것. 속도보다 가드레일이 우선입니다.

5. Bot-only 스레드 자동 해결

현재 head에서 clean rerun이 완료된 후, 봇만 참여한 스레드는 자동으로 resolve합니다. 사람이 참여한 스레드는 절대 건드리지 않습니다. 이게 개발자 경험에서 꽤 큰 차이를 만듭니다.

6. 브라우저 증거를 기계적으로 검증

UI 변경이 있으면 PR 텍스트에 스크린샷을 붙이는 게 아니라, CI에서 기계적으로 검증합니다. 필수 플로우가 존재하는지, 올바른 엔트리포인트를 사용했는지, 로그인 플로우에서 올바른 계정 identity가 있는지, artifact가 fresh한지.

7. Harness-gap loop

프로덕션에서 회귀가 발생하면 harness gap issue를 만들고, 테스트 케이스를 추가하고, SLA를 추적합니다. 일회성 핫픽스가 아니라 반복 가능한 커버리지로 전환하는 겁니다. 이게 장기적으로 가장 중요한 단계라고 생각합니다.

Carson 팀은 Greptile(리뷰)과 Codex Action(자동 수정)을 쓰지만, 이건 하나의 구현체일 뿐입니다. 패턴의 핵심은 도구가 아니라 control-plane semantics입니다. preflight gate, SHA discipline, 단일 rerun writer, 정책 게이트 우회 금지. 이 의미론만 유지하면 어떤 리뷰 에이전트든 교체할 수 있습니다.

Codex 팀의 Ryan Lopopolo는 이걸 "Harness Engineering"이라고 부릅니다. 엔지니어의 역할이 코드를 작성하는 사람에서, 에이전트가 일할 수 있는 환경을 설계하고 의도를 명세하고 피드백 루프를 만드는 사람으로 바뀌는 겁니다. Humans steer, agents execute.
