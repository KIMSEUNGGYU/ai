## 결정사항
- PARA 과잉 → 단순 5폴더 구조로 (비판 반영)
- 프로젝트당 고정 4파일 강제 X → 공통(context+decisions) + 선택(프로젝트별 자유)
- 20_me/: core.md + now.md — 시간축 분리
- identity.md 삭제: core.md가 지식, conventions.md가 행동 지시
- /done + /recap 둘 다 축적 유지 — 프로젝트가 오래 가면 /done 안 불리는 기간에 /recap이 보완
- 주입 전략: me/ 전문 + projects/ 전체 .md 로드
- hq/map.json으로 cwd→프로젝트 매핑 (PROJECT_MAP 하드코딩 제거)
- active frontmatter `context:` 로 추가 프로젝트 명시적 로드
- 플랫 구조: 프로젝트 = 독립 컨텍스트 단위 (서비스든 태스크든)
- extract-corrections.mjs: recap의 extract-transcripts.mjs 기반으로 재작성
- conventions.md: FE 규칙은 fe-workflow 플러그인으로 이동, 전역에는 도구/원칙만
- context.md 설계 원칙: 코드에서 읽을 수 있는 건 빼고, 판단에 필요한 도메인 지식만. 200줄 이하 목표
- context.md 4섹션: 서비스 정체성 + 핵심 비즈니스 흐름 + 도메인 용어 + 비표준 패턴/주의사항
- 모노레포 루트 감지: systemMessage + additionalContext 양쪽에 AskUserQuestion 지시
- /done 템플릿, /brain 스킬은 백로그로 — 실전 데이터 부족 상태에서 도구 만드는 것보다 사용하며 개선이 효율적

## 기각된 대안
- .map.json (dotfile) → 기각: Obsidian에서 안 보임. map.json으로 변경
- active 파일명 토큰 자동 매칭 → 기각: false match 리스크, 명시적 지정(frontmatter)이 예측 가능
- cwd 폴더명 자동 추론 → 기각: 이름 불일치(ishopcare-frontend ≠ ishopcare) 문제
- 프로젝트 중첩 구조(ishopcare/admin/) → 기각: 1뎁스 플랫이 조회/코드 단순
- 토큰 예산제 (프로젝트당 상한) → 기각: 인위적 제한. 양은 /brain으로 나중에 관리
- /recap에서 decisions 추출 제거 → 기각: 프로젝트 장기 유지 시 /recap이 보완 역할
- 모노레포 루트에서 전체 context 로드 → 기각: 토큰 낭비. AskUserQuestion으로 선택이 효율적
- context.md에 API 목록/폴더 트리 포함 → 기각: 코드에서 읽을 수 있는 정보는 노이즈. claude-code-guide 조사 결과 150-200줄 초과 시 compliance 급락
