# 세컨드 브레인 — 프로젝트 context.md 설계

> AI의 판단력을 높이는 프로젝트 컨텍스트 설계. "코딩 레퍼런스"가 아닌 "판단 입력".

## 배경

세컨드 브레인 Phase 2에서 각 프로젝트(admin, partners 등)의 context.md를 작성해야 한다.
이전 시도에서 퀄리티가 낮았던 원인: 코드에서 읽을 수 있는 정보(파일 수, API 목록, 폴더 트리)를 나열하는 데 그쳐 AI 판단에 실질적 도움이 안 됐음.

## 설계 원칙

### 포함해야 하는 것 (고신호)
- **도메인 비즈니스 흐름** — 코드 변수명만으로는 파악 불가능한 업무 맥락
- **도메인 용어의 비즈니스 의미** — documentTask가 "VAN사 가맹 신청"이라는 것
- **단계 전환 조건** — 수동/자동 구분, 어떤 상태에서 어떤 액션이 가능한지
- **비표준 패턴** — 프로젝트 고유 구조로 코드를 안 읽고는 모르는 것
- **주의사항** — AI가 틀리기 쉬운 함정 (직관과 다른 동작)

### 제외해야 하는 것 (저신호)
- **코드에서 읽을 수 있는 것**: API 엔드포인트 목록, 상태 enum 전체, 쿼리키 구조, 컴포넌트 목록
- **기술 스택**: package.json이나 CLAUDE.md에서 파악 가능
- **파일 수/복잡도 수치**: AI 판단에 무의미
- **폴더 트리 전체**: ls 한 번으로 파악 가능

### 토큰 예산
- 목표: **200줄 이하** (Anthropic 가이드: 150-200줄 초과 시 compliance 급락)
- context.md는 SessionStart 시 매번 전문 주입되므로 짧을수록 좋음

## context.md 구조 (4섹션)

### 1. 서비스 정체성 (2-3줄)
서비스가 뭔지 한 문단으로. 기술 스택 제외.

### 2. 핵심 비즈니스 흐름
- 메인 흐름 + 단계 전환 조건 (수동/자동 구분)
- 보완/분기 프로세스
- 서브 도메인 간 관계
- editState 같은 비즈니스 규칙

### 3. 도메인 용어
코드 변수명만으로 의미를 알 수 없는 비즈니스 용어만.
"VAN 12종"처럼 구체 목록은 코드 enum 참조로 충분.

### 4. 비표준 패턴 + 주의사항
- 비표준 패턴: 프로젝트 고유 구조, 일반적이지 않은 컨벤션
- 주의사항: 직관과 다르게 동작하는 포인트, 모르면 버그 만드는 함정

## 적용 범위

이 설계는 admin뿐 아니라 모든 프로젝트 context.md에 공통 적용:
- admin, partners, ishopcare, ishopcare-server, ai 등
- 프로젝트 특성에 따라 섹션 비중은 달라질 수 있음

### 서비스 정체성

iShopCare 내부 관리자 서비스. 카드 단말기 주문부터 VAN 가맹 완료까지의
전체 온보딩 퍼널을 관리한다. 주문→서류검수→청약→출고 4단계가 핵심 흐름.

### 핵심 비즈니스 흐름

#### 메인 퍼널 (order-document)

하나의 주문(orderNo)이 4단계를 통과한다:

주문(ORDER) → 서류검수(DOCUMENT) → 청약(DOCUMENT_TASK) → 출고(SHIPMENT)

목록은 7개 탭으로 같은 퍼널을 다른 시각에서 본다:
- 주문/i-partners/모두싸인 → 주문 단계
- 서류검수/서류보완 → 서류검수 단계
- 청약 → 청약 단계
- 출고 → 출고 단계

상세는 하나의 URL(/order-document/{orderNo})에서 ?step=0~3으로 4단계를 전환한다.
각 단계의 존재 여부는 서버가 hasOrder/hasDocument/hasDocumentTask/hasShipmentItem 플래그로 알려준다.

**단계 전환 조건:**
- 주문→서류검수: i-Partners 서류 제출 완료 또는 어드민이 수동으로 서류 추가 시 자동 생성
- 서류검수→청약: 수동 트리거. 서류 검수-완료 상태에서만 "청약 업무 생성" 버튼이 노출됨. 청약 타입을 복수 선택(QVAN-VAN, QVAN-토스페이, BizFast-가맹신규, BizFast-제신고) 후 생성
- 청약→출고: 백엔드에서 자동 생성. 프론트에서 출고 항목 생성 UI 없음

**서류 생성 3가지 경로:**
1. i-Partners: 고객에게 서류 제출 링크 발송 → 고객이 직접 작성 → 서류 자동 생성
2. 모두싸인: 전자서명 서류를 끌어와 등록 (30초~2분 소요)
3. 수동 추가: 어드민이 직접 서류 정보 + 파일 입력

creationType이 I_PARTNERS인지에 따라 보완 발송 UI가 달라진다 (테이블 입력 vs 멀티셀렉트).

**보완 프로세스:**
서류검수와 청약 모두 보완이 발생할 수 있다:
- 서류검수: 검수 진행 중 → 보완 발송 → 보완 대기 → 보완 진행 중 → 보완 확인 → 완료
- 청약: 진행 중 → 보완 요청 → 보완 대기 → 보완 진행 중 → 보완 확인 → 보완 완료

보완 발송 시 보완 사유(파일 종류별 반려 사유)를 함께 기록한다.

**서류 검수 editState 규칙:**
서류 정보 수정 가능 여부는 상태에 따라 3단계로 나뉜다:
- editable(수정 가능): 서류검수-진행중, 서류보완-대기, 서류보완-진행중
- pending(진행중으로 변경 필요): 서류검수-대기, 서류보완-확인
- readonly: 그 외 모든 상태

**담당자 자동 배정:**
서류검수와 청약 각각 독립된 담당자 풀이 있다.
담당자는 스킬(개인/법인/제신고/명의변경 x 업무타입)을 보유하고, isAssignable 플래그로 배정 대상 여부를 관리.
청약 생성 시 isAutoAssign 옵션으로 스킬 매칭 기반 자동 배정 선택 가능.

**업무 로그 (WorkLog):**
모든 단계에서 우측 스레드 패널로 업무 로그를 기록한다. orderNo 기준으로 전 스텝 통합 조회.
로그 타입: 업무 생성, 상태 변경, 담당자 할당, 알림톡 발송, 메모, 슬랙 발송 등.
@멘션으로 Slack 연동, 파일 첨부(presigned URL → S3)도 가능.
단, 상태 변경 선택은 주문(0)과 서류검수(1) 스텝에서만 가능 — 청약(2), 출고(3)에서는 비활성.

**열람 사유:**
상세 진입 시 열람 사유 입력 모달이 뜬다 (개인정보 보호). 모든 상세 페이지 공통.

#### 배달 파트너 온보딩 (delivery-status)

메인 퍼널과 별도. 단말기 구매 고객에게 배달앱(배민/요기요/쿠팡/땡겨요) 제휴를 안내하는 CX 업무.
주문 1건에 여러 배달 파트너가 연결될 수 있고, 파트너별로 독립적인 상담 상태를 가진다.
상담 상태: 신규 → 컨택중(1~3차 부재 포함) → 라이브대기 → 신청완료 → 라이브완료, 또는 취소 분기.

#### 영업 채널 (branch + dealer)

자대리점(branch)이 가맹점(merchant)을 관리하고, 딜러(dealer)가 리드(가망고객)를 영업한다.
리드 상태: 리드인입 → 상담중 → 서류대기 → 가맹심사중 → 출고대기 → 배송 → 설치완료 (또는 전환실패).
리드가 계약되면 주문이 생성되어 메인 퍼널에 진입.
자대리점에는 VAN별 코드(vanCode)와 정산률(settlementRate)이 설정된다.

#### 출고 관리 (order-shipment)

출고는 이카운트(ERP) 연동이 핵심.
상품 옵션 조합 문자열 → 이카운트 품목코드 매핑 테이블이 있고, 매핑 안 된 조합은 수동 매핑 필요.
출고 시트를 생성하면 이카운트로 전송된다 (3분 타임아웃).
출고 상태: 출고 가능 → 출고 완료/출고 불가/검수 필요/MD 필요/출고 취소.

#### 정산 (settlement)

자대리점에 대한 VAN 수수료 월별 정산.
승인분/매입분 두 유형, 세금계산서발행/소득세주민세차감 두 방식.
워크플로우: 엑셀 업로드 → 정산서 발행 → 이메일 발송.

### 도메인 용어

코드의 변수명/타입명만으로는 파악하기 어려운 비즈니스 맥락.

| 용어 | 의미 |
|------|------|
| VAN | Value Added Network. 카드 결제를 중계하는 망. 12종. 가맹점은 VAN사에 가맹 신청(청약)을 해야 카드 결제가 가능 |
| 청약 | VAN사에 카드 가맹을 신청하는 업무. 코드에서는 document-task |
| BizFast | VAN사 가맹 신청을 대행하는 외부 시스템. 신규 가맹과 제신고 두 타입 |
| QVAN | 또 다른 가맹 신청 채널. VAN 직접 신청과 토스페이 신청 두 타입 |
| 제신고 | 기존 가맹점의 사업자 정보 변경을 VAN사에 다시 신고하는 것 |
| 선출고 | 청약 완료 전에 단말기를 먼저 출고하는 방식. deliveryType 필드 |
| 가맹유형 | 신규(처음 가맹) vs 기가맹(이미 VAN 가맹이 있는 상태) |
| 모두싸인(Modusign) | 전자서명 SaaS. 계약서에 전자서명을 받을 때 사용 |
| i-Partners | 가맹점주가 직접 서류를 제출하는 웹 시스템 |
| 이카운트(ECount) | ERP. 출고 시 품목 코드와 재고를 관리 |
| 자대리점(branch) | 가맹점을 모집/관리하는 대리점. i-Agency 시스템에서 관리 |
| 딜러(dealer) | 자대리점 소속 영업 담당자. 리드(가망고객)를 관리 |
| 리드(lead) | 딜러가 관리하는 가망 고객. 계약 성사 시 주문으로 전환 |
| 라이브 | 배달앱 제휴가 완료되어 실제 서비스가 시작된 상태 |
| CX | Customer Experience. 고객 상담 이력 |

### 비표준 패턴

**pages/ vs src/pages/ 이중 구조:**
Next.js 라우팅 파일(pages/)은 단순 re-export만 한다. 실제 구현은 전부 src/pages/ 하위.

**퍼널 상세의 단일 URL + step 파라미터:**
/order-document/{orderNo}?step=0~3 으로 4단계를 하나의 URL에서 전환.
ts-pattern의 match(step).with()로 탭 컴포넌트를 분기.

**필터 상태 전부 URL (nuqs):**
모든 목록 페이지의 필터가 nuqs로 URL Query String에 동기화.
useXxxFilters 훅이 nuqs 파서를 래핑하고, SearchParamsBuilder로 API 파라미터를 생성.

**파일 업로드 2단계 (presigned URL):**
presigned-url 발급 → S3 직접 PUT 방식. 서류 첨부파일, 업무 로그 파일 모두 이 패턴.

**이중 HTTP 클라이언트:**
apiClient(인터셉터 포함)와 refreshApiClient(인터셉터 없음) 분리 운용. 토큰 갱신 무한루프 방지.

**Phase 기반 환경 분기:**
@ishopcare/phase 패키지로 local/dev/live 3단계 환경 구분. 사이드바 메뉴의 phases 필드로 환경별 노출 제어.

### 주의사항

**목록 탭 7개 != 상세 타입 4개:**
목록은 7개 탭이지만 상세 API의 type은 4개(order/document/document-task/shipment).
merchantDocuments, modusignDocuments 탭에서 상세 진입 시 step=0(order)으로 랜딩.

**creationType에 따른 UI 분기:**
서류 생성 경로(I_PARTNERS vs MODUSIGN)에 따라 보완 발송 UI가 완전히 달라진다.
I_PARTNERS: 파일 종류+사유를 테이블로 입력 / MODUSIGN: 멀티셀렉트.

**업무 로그 상태 변경은 주문/서류검수에서만:**
청약(2), 출고(3) 스텝에서는 WorkLogForm의 상태 선택이 비활성화된다.

**청약 탭 서류 섹션 숨김 상태:**
청약 상세(step=2)의 서류 상세/첨부파일 섹션이 주석 처리되어 있음 (2026.03.19~).
백엔드는 데이터를 보내지만 프론트에서 렌더링하지 않는 상태.

**모두싸인 서류 등록은 비동기:**
모두싸인 문서를 끌어오는 데 30초~2분 소요. POST 후 목록에서 새로고침해야 확인 가능.

**출고 시트 생성 타임아웃 3분:**
이카운트 연동이 포함된 출고 시트 생성/다운로드는 timeout 3분. 일반 API와 다름.
