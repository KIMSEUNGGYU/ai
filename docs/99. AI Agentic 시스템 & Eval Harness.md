# AI Agentic 시스템 & Eval Harness

## 핵심 발견

- LLM은 계획을 못 한다 → 루프(ReAct)로 속인다
- Agentic 시스템은 계획→실행→평가→재계획을 반복
- 비결정적 출력은 기존 단위테스트로 검증 불가 → Eval Harness 필요
- LLM-as-Judge 편향 문제 → Consensus(다수결)로 완화

## 상세

### A. Agentic 루프 — 어떻게 "계획"하는가

**ReAct 패턴** (Reasoning + Acting):

```
[사용자 입력]
     ↓
  Think (LLM이 다음 행동 결정)
     ↓
  Act (Tool 호출 / 코드 실행)
     ↓
  Observe (결과 관찰)
     ↓
  Think again → Act → Observe → ...
     ↓
  [목표 달성 or 실패 판단]
```

**왜 루프인가?**
LLM은 단발성 예측 기계. 긴 계획을 한 번에 세울 수 없음.
매 스텝마다 "지금 뭘 해야 하나?"를 재계산 → 복잡한 작업 처리 가능.

**Ouroboros의 Double Diamond**
ReAct를 4단계로 구체화:
- `Discover` → `Define` = Think (발산 → 수렴)
- `Design` → `Deliver` = Act (발산 → 수렴)

발산→수렴을 두 번 반복하는 이유: 첫 번째는 "무엇을 만들지" 정의, 두 번째는 "어떻게 만들지" 설계. 단계 분리로 범위 혼선 방지.

---

### C. Eval Harness — 비결정적 출력을 어떻게 평가하는가

**3단계 검증 파이프라인:**

```
출력물
  ↓
[1단계] Mechanical — "형식은 맞나?" (JSON 스키마, regex, 길이 등)
  ↓ 통과
[2단계] Semantic — "의미가 맞나?" (LLM-as-Judge)
  ↓ 통과
[3단계] Consensus — "여러 모델이 동의하는가?" (과반수 투표)
```

**왜 기존 단위테스트가 안 되는가?**

```python
# 전통 테스트: 결정적
assert add(2, 3) == 5  # 항상 같은 답

# AI 에이전트: 비결정적
assert agent.summarize(doc) == ???  # 정답이 없음
```

→ "좋은 요약"의 기준이 없음 → LLM이 LLM을 평가(LLM-as-Judge)하는 구조 필요

**LLM-as-Judge 문제점:**
- 자기 출력에 높은 점수를 주는 편향(self-preference bias) 존재
- → Consensus(다수결): 서로 다른 모델들이 동의해야 통과시켜 편향 완화

**Ouroboros의 `Ambiguity ≤ 0.2` 조건**
Eval Harness의 평가 스코어 기반. 명세서의 모호성을 수치화해서 0.2 이하일 때만 다음 단계 진행.
