## 보강 1: "무엇"이 아니라 "왜 거기에 있는가"

반복 패턴: 기능(무엇)은 맞추지만 설계 의도(왜)를 놓침.

```
Static Gate → 뭐 하는 거? "타입/린트 검증" (맞음)
             → 왜 Evaluator 전에 둔 거? "Evaluator가 Opus라서 비쌈.
               비용 0인 tsc/biome로 먼저 거르면 Opus 호출을 줄임"

B/C 평가   → 뭐 하는 거? "열린 평가 + 반대 관점" (맞음)
             → 왜 A에 추가? "A(Contract)만 보면 시야가 갇힘.
               contract에 안 쓴 패턴 불일치, 확증 편향을 잡으려고"

A/B 비교   → 뭐가 다른 거? "Plugin vs 서비스" (맞음)
             → 왜 둘 다 만듦? "격리 수준이 품질에 영향을 주는가를 실험.
               사용성은 A가 당연히 좋으니 비교 대상이 아님"
```

**공식: 설계 결정을 볼 때 "이것이 없으면 뭐가 안 되는가?"를 먼저 생각하면 "왜"가 보인다.**

## 보강 2: 수렴 감지 — 정체 vs 진동

두 번 틀린 부분. 핵심: **패턴이 다르면 대응이 다르다.**

```
┌─ 정체 (stagnation)
│  패턴: 같은 점수 반복     [7.5, 7.5, 7.5]
│  의미: 더 돌려도 안 나아짐
│  대응: 수용 (현재 결과물을 가져감)
│
├─ 진동 (oscillation)
│  패턴: N ≈ N-2 (오르내림)  [7.5, 8.0, 7.6]
│  의미: 고치면 다른 데가 망가짐
│  대응: 중단 (더 돌리면 악순환)
│
├─ 악화 (degradation)
│  패턴: fail 수 증가        [8.0, 7.5, 7.0]
│  의미: 점점 나빠짐
│  대응: pivot (방향 전환) 또는 중단
│
└─ 개선 (improving)
   패턴: 점수 상승           [7.2, 7.8, 8.1]
   의미: 잘 가고 있음
   대응: 계속 재시도
```

**구분법:**
- 3개 다 같다 → 정체
- 1번째와 3번째가 비슷한데 2번째가 다르다 → 진동
- 계속 내려간다 → 악화
- 계속 올라간다 → 개선

## 보강 3: D단계 ≠ Planner 보완

```
Planner  → spec (전체 스펙: 기능 목록, UI 구조, API 매핑, Sprint 분해)
Contract → Sprint별 완료 기준 (이번 Sprint에서 뭘 만들고, 뭘 기준으로 평가하는가)
```

D단계가 보완하는 건 **Contract**다:

```
spec: "주문서류 상세 페이지를 만든다" (전체 방향) → Planner 담당
contract: "remote 네이밍이 기존 패턴과 일관되어야 한다" (평가 기준) → D단계가 검토

D단계: "contract에 네이밍 기준이 없는데 어떻게 평가하지?" → 기준 추가 제안
```

Planner는 "무엇을 만드는가"를 정하고, Contract는 "어떻게 평가하는가"를 정한다. D단계는 후자를 보완.

## 보강 4: Agent SDK vs Anthropic SDK — 도구가 핵심

```
Agent SDK:
  callClaude("이 파일 읽고 코드 생성해")
  → Claude Code가 Read/Write/Bash 도구를 실행
  → Generator가 직접 파일을 읽고, 코드를 쓰고, tsc를 돌릴 수 있음
  → 공짜로 얻는 기능

Anthropic SDK:
  client.messages.create({ tools: [...] })
  → "파일 읽기" tool_use 요청이 오면 → 내가 readFileSync() 실행 → 결과 전달
  → "파일 쓰기" tool_use 요청이 오면 → 내가 writeFileSync() 실행 → 결과 전달
  → 이 루프를 전부 직접 구현해야 함
```

과금(구독 vs 종량)은 부차적. **Generator가 코드를 만들고 파일에 써야 하는데, Agent SDK는 이걸 Claude Code가 알아서 해준다.** Anthropic SDK는 직접 만들어야 함.

## 보강 5: 모호성 점수 — 임계값 기준

```
점수 ≤ 0.2  → 충분히 명확 → 사람에게 확인 요청
점수 > 0.2  → 아직 부족  → 어떤 차원이 부족한지 식별 → 추가 질문 → 루프

0.35의 경우:
  → 0.2보다 높으므로 아직 부족
  → "목표는 명확한데(0.1) 성공 기준이 모호하다(0.6)" 같은 진단
  → 성공 기준에 대해 추가 질문
  → 점수가 0.2 이하로 내려가면 비로소 사람에게 넘김
```

**사람에게 넘기는 건 "충분히 명확할 때"다.** 부족할 때 넘기면 사람이 애매한 스펙을 보고 판단해야 함.

## 보강 6: 파일 통신 약점 — 속도가 아니라 파싱

```
LLM 호출: 수십 초
디스크 I/O: 수 밀리초
→ 디스크 속도는 문제가 아님

진짜 약점:
  Evaluator 출력: "품질 점수: 8.1/10"
  코드 파싱:      content.match(/품질 점수:\s*([\d.]+)\/10/)

  만약 Evaluator가 "품질점수: 8.1/10" (띄어쓰기 다름) 으로 출력하면?
  → 파싱 실패 → 점수 0으로 인식 → 잘못된 판단

메모리가 아닌 json-schema가 대안인 이유:
  프로세스가 분리되어 있어서 메모리 공유가 안 됨.
  json-schema는 파일(또는 stdout)로 전달하되 형식이 보장됨.
```
