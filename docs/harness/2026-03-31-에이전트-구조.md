# AI 에이전트 시스템 — 개념과 구조

## 1. 전체 그림

하나의 LLM에게 "알아서 해"라고 하면 품질이 불안정하다. 에이전트 시스템은 이를 **역할 분리 + 루프 + 합의**로 해결한다.

```
                        Orchestrator (지휘자)
                       ┌──────┴──────┐
                       │             │
              ┌────────▼──┐    ┌─────▼─────┐
              │  Planning  │    │ Build Loop │
              │  Phase     │    │  Phase     │
              └────────────┘    └───────────┘
                   │                  │
            소크라테스식 질문     Contract → Generate → Evaluate
            + 모호성 검증        + 수렴 감지 + 진화 루프
```

---

## 2. Orchestrator — 지휘자

**LLM이 아닌 코드 로직.** if/else로 흐름을 제어한다.

왜 LLM이 아닌가: 분기 판단은 점수 비교, 횟수 카운트 같은 기계적 작업. LLM을 쓰면 비용만 늘고 판단이 불안정해진다.

```typescript
// Orchestrator가 하는 일
for (const sprint of sprints) {
  const contract = await createContract(sprint);     // 합의
  let round = 0;
  while (round < maxRetries) {
    const code = await generate(contract);            // 구현
    const gate = runStaticGate(code);                 // 정적 검증
    const eval = await evaluate(contract, code);      // 평가
    if (eval.passed) break;                           // 통과
    if (isStagnating(evalHistory)) break;             // 수렴 감지
    if (eval.contractFeedback) updateContract();      // 기준 보완
    round++;
  }
}
```

Orchestrator가 관리하는 것:
- **흐름**: 어떤 에이전트를 어떤 순서로 호출할지
- **상태**: 현재 몇 라운드인지, 점수 추이는 어떤지
- **분기**: 통과/재시도/중단 판단
- **파일**: 에이전트 간 주고받는 문서(spec, contract, feedback) 저장

## 3. Planning Phase — 소크라테스식 질문 + 모호성 검증

단순히 "스펙 만들어"가 아니라, **숨겨진 가정을 드러내는 루프**를 돌린다.

### 소크라테스식 질문

사용자가 말하지 않은 것을 찾는다:

```
사용자: "주문 취소 페이지 만들어"
Planner: "취소하면 환불도 자동 처리되나요, 별도 프로세스인가요?"
Planner: "취소 사유는 고정 선택지인가요, 자유 입력인가요?"
Planner: "이미 배송 중인 주문도 취소 가능한가요?"
```

왜 필요한가: 요구사항의 80%는 말하지 않은 가정에 있다. "취소 페이지"라고 했지만 환불 정책, 상태 분기, 권한 체크 등이 숨어 있음.

### 모호성 자체 점검 (알고리즘)

스펙을 만든 후 Planner가 **자기 결과물을 정량 평가**한다:

```
모호성 점수 = 1 - Σ(clarityᵢ × weightᵢ)

| 차원        | 가중치 | 체크                                    |
|------------|--------|----------------------------------------|
| 목표 명확도 | 40%   | 기능이 구체적인가? "등등" 없는가?         |
| 제약 명확도 | 30%   | API, 기존 코드 재사용이 명시되었는가?     |
| 성공 기준   | 30%   | Sprint별 산출물과 범위가 명확한가?        |
```

- 점수 > 0.2 → 부족한 차원을 식별하고 사용자에게 추가 질문
- 점수 ≤ 0.2 → 사람에게 확인 요청

이 루프의 핵심: **"충분히 명확한가?"를 감으로 판단하지 않고 알고리즘으로 판단**한다.

```
요구사항 입력
  ↓
소크라테스식 질문 (숨겨진 가정 발견)
  ↓
스펙 확장 ("무엇"만 정의, "어떻게"는 Generator에게)
  ↓
모호성 점수 계산
  ↓ ≤ 0.2?
  ├─ No → 부족한 차원 식별 → 추가 질문 → (루프)
  └─ Yes → 사람 확인 → 확정 → spec.md 출력
```

## 4. Contract — 만들기 전에 합의

Generator와 Evaluator가 코드를 작성하기 **전에** "뭘 만드는지"와 "뭘 기준으로 평가하는지"를 맞춘다.

### 왜 필요한가

contract 없이 진행하면:
- Generator: "내가 판단해서 이렇게 만들었다" (범위 초과, 임의 판단)
- Evaluator: "뭘 기준으로 평가하지?" (기준 애매, 주관적 평가)

contract가 있으면:
- Generator: "contract에 있는 것만 만든다" (범위 엄수)
- Evaluator: "contract 항목을 하나씩 pass/fail 한다" (객관적 평가)

### 구조

```markdown
# Sprint 1 Contract

## 이번 Sprint에서 만드는 것        ← Generator의 할 일
## 하지 말아야 할 것                ← 범위 밖 명시 (중요)
## 완료 기준 (Evaluator가 검증)     ← 평가 기준
## 참조할 기존 코드                 ← 패턴 따를 대상
```

"하지 말아야 할 것"이 중요한 이유: LLM은 범위를 넘기는 경향이 있다. "UI는 다음 Sprint"라고 명시해야 Generator가 자제함.

### 생성 과정

```
Planner가 초안 작성
  ↓
Evaluator가 검토 ("이 기준으로 평가 가능한가?")
  ↓
보완 후 합의 → contract.md
```

---

## 5. Build Loop — Generate → Evaluate → 진화

contract가 합의되면 AI가 자율적으로 루프를 돌린다.

### 루프 구조

```
        ┌──────────────────────────────────────┐
        │                                      │
        ▼                                      │
   [Generator]  contract + feedback로 구현      │
        │                                      │
        ▼                                      │
   [Static Gate]  tsc + biome (비용 0)          │
        │ 실패 → 에러 전달 → Generator 재호출    │
        ▼                                      │
   [Evaluator]  4단계 평가                      │
        │                                      │
        ▼                                      │
   ┌─ PASS → 다음 Sprint                       │
   │                                           │
   └─ FAIL → 수렴 감지 판단 ──────────────────→ │
              │                                │
              ├─ 개선 중 → feedback + contract 보완 → 재시도
              ├─ 정체 → 수용
              ├─ 악화 → pivot 또는 중단
              └─ hard cap → 중단
```

### Static Gate — 비용 0 필터

Evaluator(Opus)를 호출하면 비용이 크다. 그 전에 기계적 검증으로 걸러냄:

```
tsc --noEmit  → 타입 에러 0?
biome check   → 린트 에러 0?
```

통과한 코드만 Evaluator에게 보내서 비용 절약.

### Evaluator 4단계 — 갇히지 않는 평가

| 단계 | 성격 | 질문 | 왜 필요한가 |
|------|------|------|-----------|
| A. Contract 기준 | 닫힌 | "약속한 거 다 했나?" | 객관적 pass/fail |
| B. 열린 평가 | 열린 | "패턴이 일관된가? 복잡도는?" | contract에 없는 문제 발견 |
| C. Contrarian | 반대 | "가장 약한 점은? 6개월 후 문제는?" | 확증 편향 방지 |
| D. Contract 검토 | 메타 | "기준 자체가 부실한가?" | 기준 누락 감지 |

A만 있으면 contract에 쓴 것만 봄 (닫힌 시야).
B를 추가하면 contract 밖 문제를 잡음.
C를 추가하면 "잘 됐다"에 강제로 의심.
D를 추가하면 코드가 아니라 기준이 문제일 때를 감지.

```
점수 산출:
  품질 점수 = (A 통과율 × 0.6) + (B × 0.3) + (C × 0.1)
  통과 조건 = A 전부 pass AND 점수 ≥ 8.0/10
```

### 수렴 감지 — Ralph 스타일

무한 루프를 방지하는 알고리즘. 점수 추이(history)를 보고 판단:

```
evalHistory = [7.2, 7.8, 8.1]  → 개선 중 → 재시도
evalHistory = [7.5, 7.5, 7.5]  → 3회 동점 → 정체로 수용
evalHistory = [7.5, 8.0, 7.6]  → N ≈ N-2 → 진동 감지 → 중단
evalHistory = [8.0, 7.5, 7.0]  → fail 증가 → 악화 → pivot/중단
```

"그냥 N번 돌리고 끝"이 아니라, **추이를 보고 의미 있는 재시도만** 한다.

### D단계 + Contract 보완 — 진화 루프

Ouroboros에서 가져온 개념. 코드만 고치는 게 아니라 **기준도 진화**한다:

```
Round 1:
  Generator: 코드 생성
  Evaluator: FAIL — "remote 네이밍 잘못됨"
             D단계: "contract에 네이밍 기준이 누락됨"
  → contract에 "네이밍 기준" 추가

Round 2:
  Generator: 보완된 contract 기반으로 코드 수정
  Evaluator: 보완된 기준으로 평가 → PASS
```

기존 루프: 코드만 수정 → 기준 누락이면 영원히 못 잡음
진화 루프: 코드 + 기준 모두 수정 → 근본 원인 해결

## 6. 에이전트 간 통신 — 파일 기반 + 정보 차단

### 파일로 소통하는 이유

```
Planner  ──writes──→  spec.md      ──reads──→  Generator, Evaluator
Planner  ──writes──→  contract.md  ──reads──→  Generator, Evaluator
Generator ──writes──→  [코드 파일]  ──reads──→  Evaluator
Evaluator ──writes──→  eval-log.md ──reads──→  Orchestrator (점수 파싱)
Evaluator ──writes──→  feedback.md ──reads──→  Generator (다음 라운드)
```

왜 파일인가:
- **사람이 개입 가능**: spec.md를 열어서 직접 수정할 수 있음
- **로그 자동 축적**: eval-log가 라운드마다 쌓여서 추이 분석 가능
- **디버깅**: "3라운드에서 뭐가 잘못됐지?" → 해당 파일을 열면 됨

### GAN 원리 — 의도적 정보 차단

```
Generator ──×──→ Evaluator의 평가 근거 (왜 이렇게 판단했는지)
Evaluator ──×──→ Generator의 사고 과정 (왜 이렇게 만들었는지)
```

Generator가 Evaluator의 판단 근거를 알면 → 평가 통과만을 위한 코드를 만듦 (게이밍)
Evaluator가 Generator의 의도를 알면 → "아마 이런 의도겠지"라고 관대해짐 (편향)

**서로의 추론을 모르기 때문에 견제가 작동한다.** Generator는 코드 품질로, Evaluator는 기준으로 승부.

## 7. 파일 관리

```
.ai/harness/
├── {도메인}/
│   ├── domain-context.md          ← 도메인 공유 설계 (DTO, API 등)
│   ├── {페이지}/                  ← 실행 단위 = 페이지 1개
│   │   ├── spec.md                ← Planning 결과
│   │   ├── sprint-1/
│   │   │   ├── contract.md        ← 합의 (보완 시 append됨)
│   │   │   ├── eval-log-r1.md     ← 평가 기록
│   │   │   ├── feedback-r1.md     ← Generator에게 전달하는 피드백
│   │   │   └── eval-log-r2.md
│   │   └── summary.md             ← 전체 실행 요약
```

- **domain-context.md**: 같은 도메인의 페이지 간 공유 설계. Sprint 완료 시 업데이트. 다음 페이지 구현 시 참조.
- **contract.md**: Evaluator가 D단계에서 기준 보완하면 append됨. 진화 이력이 남음.
- **summary.md**: Sprint별 점수, 라운드 수, 미해결 사항. 사람이 최종 판단할 때 사용.

## 8. 성과 추적

### Sprint 단위

```
| Sprint | 범위      | 라운드 | 최종 점수 | 결과       |
|--------|----------|--------|----------|-----------|
| 1      | API 계층  | 2      | 9.1/10   | pass      |
| 2      | UI       | 1      | 8.5/10   | pass      |
| 3      | 인터랙션  | 3      | 7.2/10   | 정체로 수용 |
```

### 라운드 단위 (eval-log)

```
Round 1: 7.2/10 — Contract 4/5 pass, 열린 평가 이슈 2건
Round 2: 8.1/10 — Contract 5/5 pass, 이슈 1건 → 개선 중
Round 3: 9.1/10 — 전부 pass → 통과
```

이 데이터가 축적되면:
- "어떤 Sprint 유형이 재시도가 많은가?" → 프롬프트 튜닝 방향
- "어떤 컨벤션 위반이 반복되는가?" → contract 기본 항목에 추가
- "Plugin vs 서비스 품질 차이는?" → A/B 비교

## 9. 사람 개입 시점

에이전트가 자율적으로 돌지만, 사람이 개입하는 시점이 정해져 있다:

```
[Planning]  사람 + AI 협업
  → 소크라테스식 질문에 답변
  → spec.md 확인/수정

[Build Loop]  AI 자율
  → 사람은 안 봐도 됨
  → 비정상 종료 시에만 알림 ("여기서 막혔다")

[완료 후]  사람 검토
  → summary.md 확인
  → 작은 수정 → 일반 Claude Code
  → 방향 변경 → 하네스 재호출
```

핵심: **계획은 같이, 실행은 AI가, 검토는 사람이.**

## 10. 설계 과정에서의 고민과 결정

### 모델 호출 방식 — 4가지 중 왜 Agent SDK인가

코드에서 Claude를 호출하는 방법이 4가지다:

```
Plugin (Agent 도구)    — Claude Code 안에서. 코드 제어 불가.
claude -p (CLI)        — 프로세스 스폰. 구독. 블로킹, 1회용.
Claude Agent SDK       — async 스트리밍. 구독. 세션 resume. ← 채택
Anthropic SDK (API)    — 완전 제어. 토큰 종량. 도구 직접 구현.
```

처음 계획은 `claude -p`였다. 구현 후 문제를 발견:
- 매 호출마다 프로세스 스폰 오버헤드
- 멀티턴 불가 (1회 호출)
- morning-brief가 이미 Agent SDK를 쓰고 있어서 불일관

Agent SDK는 `claude -p`의 상위 호환이다. `callClaude` 함수 하나만 교체하면 되서 전환 비용도 낮았다.

Anthropic SDK(API)는 검토했지만 탈락:
- Generator가 파일을 직접 써야 하는데, tool_use 핸들러를 전부 구현해야 함
- Agent SDK는 Claude Code 도구(Read, Write, Bash)를 공짜로 쓸 수 있음
- 토큰 종량 과금이 부담

**결정: 구독 과금 + 코드 제어 + Claude Code 도구 활용 = Agent SDK.**

상세 비교: `docs/Claude 모델 호출 방식 비교.md`

---

### Ouroboros에서 뭘 가져오고 뭘 버렸는가

Ouroboros(Q00/ouroboros)는 진화형 코드 생성 엔진. 분석 후 4가지를 검토:

**가져온 것: 루프 개선 (Contract 진화)**

Ouroboros의 핵심 패턴: Wonder("아직 모르는 게 뭐지?") → Reflect(스펙 수정) → 다음 세대.
코드만 고치는 게 아니라 기준/스펙 자체를 진화시킨다.

우리에게 적용: Evaluator D단계(Contract 검토) 추가. "기준이 빠져있다"를 감지하면 contract를 보완.
변경 최소(프롬프트 + 파싱 + Orchestrator 분기 추가), 효과 큼(근본 원인 해결 가능).

**버린 것 3가지:**

1. **병렬 실행** — Ouroboros는 AC(Acceptance Criteria) 간 의존성을 분석해서 독립적인 것을 동시 실행.
   버린 이유: FE Sprint은 의존성이 높다. Sprint 2(UI)가 Sprint 1(API)의 DTO를 import하는 게 일반적. 독립 Sprint이 드물어서 효과 제한적.

2. **메모리 통신** — Ouroboros는 Python 객체로 에이전트 간 직접 전달.
   버린 이유: 우리는 프로세스가 분리(SDK query()가 별도 프로세스)되어 메모리 공유가 안 됨. 파일이 자연스러운 인터페이스. 사람이 중간 결과를 직접 읽을 수 있는 장점도 있음.

3. **SQLite 영속화** — Ouroboros는 이벤트 소싱 패턴으로 SQLite에 기록.
   버린 이유: 분석할 데이터가 아직 없다. eval-log를 사람이 직접 읽는 단계. SQLite는 "50회 실행의 평균 점수"같은 쿼리가 필요해질 때 추가하면 됨. 지금은 관리 포인트만 늘어남.

**판단 기준: "지금 이 규모에서 효과가 있는가?"** 병렬/SQLite는 오버엔지니어링, 루프 개선은 변경 최소 + 효과 큼.

---

### Plugin(A) vs 독립 서비스(B) — 왜 둘 다 만드는가

같은 설계를 두 방식으로 구현하는 건 비효율적으로 보인다. 이유는 하나:

**"컨텍스트 격리가 실제로 품질에 영향을 주는가?"를 실험하기 위해.**

가설: Generator가 Evaluator의 사고를 볼 수 없으면(완전 격리) → Self-Evaluation Bias가 줄어서 → 결과물 품질이 올라갈 것.

```
A (Plugin):   같은 세션 → Generator가 Evaluator 컨텍스트를 일부 볼 수 있음
B (서비스):   별도 프로세스 → 완전 격리
```

A로 충분하면 B는 불필요. B가 확실히 좋으면 불편해도 B를 써야 함. **써보고 결정.**

---

### 통신 방식 — 파일의 약점과 대안

파일 기반 통신의 현재 약점: **파싱이 취약하다.**

```typescript
// scoring.ts — eval-log에서 점수를 정규식으로 추출
const scoreMatch = content.match(/품질 점수:\s*([\d.]+)\/10/);
```

Evaluator가 형식을 조금만 틀어도 파싱 실패. 지금은 동작하지만 프롬프트 튜닝 과정에서 깨질 수 있다.

대안: SDK의 `json-schema` 옵션으로 **코드용(JSON) + 사람용(마크다운) 분리.**
- Evaluator가 구조화된 JSON 출력 → 코드가 안정적으로 파싱
- 별도로 eval-log.md를 생성 → 사람이 읽을 수 있음

지금 바꾸지 않는 이유: 아직 실제로 깨진 적이 없다. 파싱 깨지는 사례가 나오면 그때 전환.

---

### AI에게 좋은 통신 형식은?

AI 입장에서는 마크다운이 자연스럽다:
- "네이밍이 fetch 패턴과 불일치 (참조: remotes/merchant.ts)" 같은 풍부한 피드백을 자연스럽게 생성
- JSON으로 강제하면 뉘앙스가 구조에 갇힘

문제는 AI가 아니라 **코드가 AI 출력을 읽을 때.** 점수, pass/fail 같은 기계적 값을 정규식으로 추출하는 게 취약한 거지, AI가 마크다운을 읽고 쓰는 건 문제없음.

결론: AI의 자연어 출력을 존중하되, 코드가 읽어야 하는 부분만 구조화하는 게 최선.

### 보강 2: 수렴 감지 — 정체 vs 진동

두 번 틀린 부분. 핵심: **패턴이 다르면 대응이 다르다.**

```
┌─ 정체 (stagnation)
│  패턴: 같은 점수 반복     [7.5, 7.5, 7.5]
│  의미: 더 돌려도 안 나아짐
│  대응: 수용 (현재 결과물을 가져감)
│
├─ 진동 (oscillation)
│  패턴: N ≈ N-2 (오르내림)  [7.5, 8.0, 7.6]
│  의미: 고치면 다른 데가 망가짐
│  대응: 중단 (더 돌리면 악순환)
│
├─ 악화 (degradation)
│  패턴: fail 수 증가        [8.0, 7.5, 7.0]
│  의미: 점점 나빠짐
│  대응: pivot (방향 전환) 또는 중단
│
└─ 개선 (improving)
   패턴: 점수 상승           [7.2, 7.8, 8.1]
   의미: 잘 가고 있음
   대응: 계속 재시도
```

**구분법:**
- 3개 다 같다 → 정체
- 1번째와 3번째가 비슷한데 2번째가 다르다 → 진동
- 계속 내려간다 → 악화
- 계속 올라간다 → 개선

### 보강 3: D단계 ≠ Planner 보완

```
Planner  → spec (전체 스펙: 기능 목록, UI 구조, API 매핑, Sprint 분해)
Contract → Sprint별 완료 기준 (이번 Sprint에서 뭘 만들고, 뭘 기준으로 평가하는가)
```

D단계가 보완하는 건 **Contract**다:

```
spec: "주문서류 상세 페이지를 만든다" (전체 방향) → Planner 담당
contract: "remote 네이밍이 기존 패턴과 일관되어야 한다" (평가 기준) → D단계가 검토

D단계: "contract에 네이밍 기준이 없는데 어떻게 평가하지?" → 기준 추가 제안
```

Planner는 "무엇을 만드는가"를 정하고, Contract는 "어떻게 평가하는가"를 정한다. D단계는 후자를 보완.

### 보강 4: Agent SDK vs Anthropic SDK — 도구가 핵심

```
Agent SDK:
  callClaude("이 파일 읽고 코드 생성해")
  → Claude Code가 Read/Write/Bash 도구를 실행
  → Generator가 직접 파일을 읽고, 코드를 쓰고, tsc를 돌릴 수 있음
  → 공짜로 얻는 기능

Anthropic SDK:
  client.messages.create({ tools: [...] })
  → "파일 읽기" tool_use 요청이 오면 → 내가 readFileSync() 실행 → 결과 전달
  → "파일 쓰기" tool_use 요청이 오면 → 내가 writeFileSync() 실행 → 결과 전달
  → 이 루프를 전부 직접 구현해야 함
```

과금(구독 vs 종량)은 부차적. **Generator가 코드를 만들고 파일에 써야 하는데, Agent SDK는 이걸 Claude Code가 알아서 해준다.** Anthropic SDK는 직접 만들어야 함.

### 보강 5: 모호성 점수 — 임계값 기준

```
점수 ≤ 0.2  → 충분히 명확 → 사람에게 확인 요청
점수 > 0.2  → 아직 부족  → 어떤 차원이 부족한지 식별 → 추가 질문 → 루프

0.35의 경우:
  → 0.2보다 높으므로 아직 부족
  → "목표는 명확한데(0.1) 성공 기준이 모호하다(0.6)" 같은 진단
  → 성공 기준에 대해 추가 질문
  → 점수가 0.2 이하로 내려가면 비로소 사람에게 넘김
```

**사람에게 넘기는 건 "충분히 명확할 때"다.** 부족할 때 넘기면 사람이 애매한 스펙을 보고 판단해야 함.

### 보강 6: 파일 통신 약점 — 속도가 아니라 파싱

```
LLM 호출: 수십 초
디스크 I/O: 수 밀리초
→ 디스크 속도는 문제가 아님

진짜 약점:
  Evaluator 출력: "품질 점수: 8.1/10"
  코드 파싱:      content.match(/품질 점수:\s*([\d.]+)\/10/)

  만약 Evaluator가 "품질점수: 8.1/10" (띄어쓰기 다름) 으로 출력하면?
  → 파싱 실패 → 점수 0으로 인식 → 잘못된 판단

메모리가 아닌 json-schema가 대안인 이유:
  프로세스가 분리되어 있어서 메모리 공유가 안 됨.
  json-schema는 파일(또는 stdout)로 전달하되 형식이 보장됨.
```
