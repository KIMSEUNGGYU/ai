# FE 하네스 — 개념 × 구현 × 설계 고민

AI 에이전트 + 하네스 시스템을 직접 구축하며 정리한 문서.
각 컴포넌트마다 **개념**(이게 뭔가) → **구현**(실제로 어떻게 만들었나) → **고민**(왜 이렇게 했나)을 함께 정리.

---

## 전체 흐름 한눈에

```
사용자: "주문서류 상세 페이지 만들어"
                    │
         ┌──────────▼──────────┐
         │   Orchestrator      │
         │   (흐름 제어)        │
         └──────────┬──────────┘
                    │
    ┌───────────────▼────────────────┐
    │  Phase 1: Planning             │
    │                                │
    │  Planner (Opus)                │
    │  ├─ 소크라테스식 질문           │
    │  ├─ 스펙 확장                  │
    │  ├─ 모호성 점수 ≤ 0.2?         │
    │  │  └─ No → 추가 질문 (루프)   │
    │  └─ spec.md 출력               │
    │                                │
    │  [사람 확인] → 확정             │
    └───────────────┬────────────────┘
                    │
    ┌───────────────▼────────────────┐
    │  Phase 2: Build Loop           │
    │  (Sprint마다 반복)              │
    │                                │
    │  ① Contract 합의               │
    │     Planner 초안 → Evaluator 검토 → contract.md
    │                                │
    │  ② Generator (Opus) 구현       │
    │     contract + spec + feedback → 코드
    │                                │
    │  ③ Static Gate (비용 0)         │
    │     tsc + biome → 실패 시 ②로  │
    │                                │
    │  ④ Evaluator (Opus) 4단계 평가 │
    │     A(Contract) + B(열린) + C(Contrarian) + D(Contract 검토)
    │     → eval-log-rN.md           │
    │                                │
    │  ⑤ 수렴 감지 (Ralph)            │
    │     점수 너무 낮음 → pivot      │
    │     개선폭 미미 → accept        │
    │     개선폭 큼 → 재시도 준비:    │
    │       D단계 피드백 있으면 contract 보완
    │       feedback → Generator에게 전달
    │       → ② 재시도               │
    └───────────────┬────────────────┘
                    │
    ┌───────────────▼────────────────┐
    │  Phase 3: Summary              │
    │  summary.md → 사람에게 전달     │
    └────────────────────────────────┘
```

- **Sprint**: Planner가 spec.md에서 분해한 작업 단위. 의존성 순서대로 순차 실행.
  - 예) Sprint 1: API 계층(DTO, remote, query) → Sprint 2: UI 컴포넌트 → Sprint 3: 인터랙션
- **루프가 두 겹**: 바깥 루프(Sprint 순차)와 안쪽 루프(Round 반복).
- Sprint 1이 완료되어야 Sprint 2로 넘어감 (Sprint 2가 Sprint 1의 산출물을 사용하므로).

**실제 동작 예시:**

```
Sprint 1: API 계층
  │
  ├─ Round 1:
  │    Generator → 코드 생성
  │    Static Gate → tsc/biome 통과
  │    Evaluator → FAIL (7.2/10, remote 네이밍 틀림)
  │    수렴 감지 → 첫 라운드, 계속
  │    contract 보완 (D단계: "네이밍 기준 누락")
  │    feedback → Generator에게 전달
  │
  ├─ Round 2:
  │    Generator → feedback 보고 코드 수정
  │    Static Gate → 통과
  │    Evaluator → FAIL (8.0/10, query 패턴 불일치)
  │    수렴 감지 → Δ0.8 (큼), 계속
  │    feedback → Generator에게 전달
  │
  ├─ Round 3:
  │    Generator → feedback 보고 수정
  │    Static Gate → 통과
  │    Evaluator → PASS (8.5/10)
  │    → Sprint 1 완료 ✓
  │
Sprint 2: UI 컴포넌트
  ├─ Round 1: ...
```

이하 각 컴포넌트를 상세 설명.

## 1. Orchestrator — 지휘자

### 개념

여러 에이전트를 조율하는 중앙 컨트롤러. 오케스트라의 지휘자처럼 "누가 언제 연주할지"를 결정한다. 에이전트가 직접 다음 에이전트를 부르는 게 아니라, Orchestrator가 흐름을 통제.

### 구현

**두 가지 버전으로 구현:**

```
A. Plugin (commands/harness.md)
   → 마크다운 프롬프트. Claude가 읽고 분기를 판단.
   → "eval-log에서 PASS/FAIL을 확인하고, FAIL이면 이전 라운드와 비교해서..."

B. 독립 서비스 (src/orchestrator.ts)
   → TypeScript 코드. if/else로 분기.
   → if (evalResult.passed) break; if (convergence.action === 'accept') break;
```

**Orchestrator가 제어하는 흐름:**

```
Phase 1: Planning   → Planner 호출 → spec.md → 사람 확인
Phase 2: Build Loop → Sprint마다 { Contract → Generate → Static Gate → Evaluate → 수렴 감지 }
Phase 3: Summary    → 결과 종합 → summary.md
```

### 고민: 왜 LLM 자유 판단이 아닌 규칙 기반인가

"LLM이 아닌 코드"란 **LLM에게 분기를 자유 판단시키지 않는다**는 뜻. Plugin(A)이든 서비스(B)든 동일한 원칙:

```
A. Plugin (harness.md):
   마크다운으로 규칙을 명시 → Claude가 규칙을 읽고 따름
   "eval-log에서 PASS면 다음 Sprint, FAIL이면 이전 라운드와 비교해서..."

B. 서비스 (orchestrator.ts):
   TypeScript로 규칙을 구현
   if (evalResult.passed) break;
```

둘 다 "규칙 기반 분기"다. 차이는 규칙을 마크다운으로 쓰느냐(A), 코드로 쓰느냐(B)일 뿐.

LLM에게 "어떻게 할지 알아서 판단해"라고 하지 않는 이유:
- "점수가 8.0 이상인가?" 같은 분기는 기계적 비교. LLM이 판단하면 불안정 ("8.0이면... 음... 통과로 볼 수도...")
- 같은 입력에 다른 분기를 탈 수 있음
- Orchestrator에는 "판단"이 아니라 "규칙"이 필요

## 2. Planner — 기획자

### 개념

한 줄 요구사항을 전체 스펙으로 확장하는 에이전트. "주문 취소 페이지 만들어" → 기능 목록, UI 구조, API 매핑, Sprint 분해까지.

핵심은 **소크라테스식 질문**: 사용자가 말하지 않은 가정을 드러냄.

```
"취소하면 환불도 자동인가요?"
"이미 배송 중인 주문도 취소 가능한가요?"
```

### 구현

```
A. Plugin: agents/planner.md (Opus)
   → 소크라테스식 질문 + 스펙 확장 + 모호성 자체 점검
   → AskUserQuestion으로 사용자와 대화

B. 서비스: src/agents/planner.ts
   → systemPrompt에 역할/원칙/컨벤션 주입
   → Claude Agent SDK query()로 호출
```

**Planner의 두 가지 역할:**

1. spec.md 생성 (Planning Phase)
2. contract.md 초안 생성 (Build Loop에서)

### 고민: "무엇"만 정의하고 "어떻게"를 안 정하는 이유

Planner가 "React Query로 구현" 같은 기술 디테일을 정하면, 틀렸을 때 전체로 퍼짐. "주문 조회 API가 필요하다"(무엇)만 정의하면 Generator가 기존 패턴을 보고 최적의 방법을 판단.

### 고민: 모호성 점수 알고리즘

감으로 "충분한가?"를 판단하지 않고 정량화:

```
모호성 점수 = 1 - Σ(clarityᵢ × weightᵢ)
| 목표 명확도 40% | 제약 명확도 30% | 성공 기준 30% |

> 0.2 → 부족한 차원 식별 → 추가 질문 (루프)
≤ 0.2 → 사람에게 확인 요청
```

"사람에게 넘기는 건 충분히 명확할 때." 부족할 때 넘기면 사람이 애매한 스펙을 보고 판단해야 함.

## 3. Contract — 사전 합의

### 개념

코드 작성 **전에** Generator와 Evaluator가 따르는 합의. "뭘 만드는지"와 "뭘 기준으로 평가하는지"를 Sprint마다 정한다. Planner가 초안을 잡고, Evaluator가 "이 기준으로 평가 가능한가?"를 검증한다. Generator는 기준 설정에 참여하지 않는다 — 참여하면 평가 통과를 위한 코드를 만들게 됨(게이밍 리스크, GAN 정보 차단 원칙).

### 구현: contract.md

```markdown
# Sprint 1 Contract: API 계층

## 이번 Sprint에서 만드는 것       ← Generator의 할 일
- OrderDocumentDetailResponse DTO
- fetchOrderDocumentDetail remote

## 하지 말아야 할 것               ← 범위 밖 명시
- UI 컴포넌트 (Sprint 2에서)

## 완료 기준 (Evaluator가 검증)    ← 평가 기준
- [ ] DTO 필드가 서버 스펙과 일치
- [ ] remote 네이밍이 기존 패턴과 일관

## 참조할 기존 코드               ← 패턴 따를 대상
- remotes/merchant.ts
```

**생성 과정:**

```
Planner가 초안 → Evaluator가 검토("이 기준으로 평가 가능한가?") → 합의
```

**진화 (D단계):** Evaluator가 기준 누락을 발견하면 contract에 "보완 기준" append:

```
## 보완 기준 (Round 1에서 추가)
- remote 네이밍이 fetch* 패턴을 따를 것
```

### 고민: "하지 말아야 할 것"이 중요한 이유

LLM은 범위를 넘기는 경향이 있음. "API만 만들어"라고 해도 "UI도 좀 만들어놓을게"가 나옴. "하지 말 것: UI 컴포넌트"를 명시해야 Generator가 자제.

### 고민: 왜 contract가 없으면 안 되는가

```
contract 없이:
  Generator: "내 판단으로 이렇게 만들었다" (범위 초과, 임의 판단)
  Evaluator: "뭘 기준으로 평가하지?" (기준 애매, 주관적 평가)

contract 있으면:
  Generator: "contract에 있는 것만 만든다"
  Evaluator: "contract 항목을 하나씩 pass/fail"
```

## 4. Generator — 구현자

### 개념

Contract 기준으로 코드를 생성하는 에이전트. Opus 모델 사용 (구독이라 비용 동일, 품질 우선. rate limit 문제 시 Sonnet으로 전환).

### 구현

```
A. Plugin: agents/generator.md (Opus)
   → disallowedTools: Bash, NotebookEdit (Read, Write, Edit 사용 가능)
   → 직접 파일을 읽고 코드를 씀

B. 서비스: src/agents/generator.ts
   → systemPrompt에 원칙/컨벤션 주입
   → Claude Agent SDK로 호출 → CC 도구로 파일 직접 작성
```

**Generator의 핵심 원칙:**

1. contract에 없는 건 안 만듦 (Sprint 범위 엄수)
2. 기존 코드 패턴을 따라감 (자기 판단으로 새 패턴 안 만듦)
3. 자기검증 안 함 ("내 코드가 잘 됐는지"는 Evaluator의 역할)
4. 재실행 시 feedback.md만 보고 수정 (이전 자기 사고 과정은 못 봄 — 프로세스 격리)

### 고민: Evaluator의 사고를 모르는 이유 (GAN 원리)

Generator가 Evaluator의 판단 근거를 알면 → 평가 통과만을 위한 코드를 만듦 (게이밍). 코드 품질이 아니라 "통과 기술"이 발달.

서로의 추론을 모르기 때문에 Generator는 코드 품질로, Evaluator는 기준으로 승부. 이것이 GAN(Generative Adversarial Network) 영감의 핵심.

## 5. Static Gate — 비용 0 필터

### 개념

Generator가 코드를 생성한 직후, Evaluator 호출 **전에** 기계적 검증으로 거르는 단계. 타입 에러가 있는 코드를 비싼 Opus에게 보낼 필요 없음.

### 구현

```
A. Plugin: tsc --noEmit + biome check + harness-check.sh (FE 컨벤션 기계적 검증)
B. 서비스: tsc --noEmit + biome check (harness-check.sh는 Plugin 전용)
```

```
Generator → 코드 생성
  ↓
Static Gate (tsc + biome)
  ├─ 통과 → Evaluator에게 보냄
  ├─ 실패 → 에러 메시지를 Generator에게 전달 → 재생성 (max 3회)
  └─ 3회 실패 → Sprint 중단
```

### 고민: 왜 Evaluator 전에 두는가

Evaluator(Opus) 호출 비용이 큼. tsc/biome는 비용 0 + 즉시 실행. 타입 에러 같은 기본적인 문제는 비싼 LLM이 볼 필요 없음.

## 6. Evaluator — 평가자

### 개념

Static Gate를 통과한 코드를 독립적으로 평가하는 에이전트. Opus 모델 사용 (판단 품질이 중요). Generator의 사고 과정을 모르고, 코드와 contract만 봄.

### 구현

```
A. Plugin: agents/evaluator.md (Opus)
   → disallowedTools: Write, Edit, Bash, NotebookEdit (읽기만 가능)
   → Read, Glob, Grep으로 코드를 직접 확인

B. 서비스: src/agents/evaluator.ts
   → systemPrompt에 컨벤션 + 점수 공식 + 통과 기준 주입
```

### 4단계 평가

```
A. Contract 기준 (닫힌)     "약속한 거 다 했나?"        → pass/fail
B. 열린 평가 (자유)          "패턴 일관? 복잡도는?"      → 시야 확장
C. Contrarian (반대)         "가장 약한 점은?"           → 확증 편향 방지
D. Contract 검토 (메타)      "기준 자체가 빠져있나?"     → 기준 진화
```

**왜 A만으로 부족한가:**

- A만 있으면 contract에 쓴 것만 봄 (시야가 갇힘)
- B가 contract 밖 문제를 잡음 (패턴 불일치 등)
- C가 "잘 됐다"에 강제 의심 (확증 편향 방지)
- D가 코드가 아니라 기준이 문제일 때를 감지 (Ouroboros에서 영감)

### 점수 체계

```
품질 점수 = (A 통과율 × 0.6) + (B × 0.3) + (C × 0.1)
통과 조건 = A 전부 pass AND 품질 점수 ≥ 8.0/10
```

### 출력 파일

**eval-log-rN.md** — 평가 기록 (라운드마다 별도 파일):

```markdown
# Sprint 1 Eval — Round 1
## 통과 판정: FAIL (Contract 미충족)
## 품질 점수: 8.1/10
## A. Contract 기준 (pass/fail 표)
## B. 열린 평가
## C. Contrarian
## D. Contract 검토
## 방향성
```

**feedback-rN.md** — Generator에게 전달하는 피드백 (FAIL 시에만):

```markdown
# Sprint 1 Feedback — Round 1
## 수정 필요
## 검토 권장
## Contract 수정 제안
```

### 고민: eval-log를 라운드마다 별도 파일로 저장하는 이유

하나의 파일에 append하면 이전 평가가 다음 평가에 영향을 줄 수 있음. 별도 파일이면 각 라운드가 독립적. 수렴 감지는 Orchestrator가 파일들을 비교해서 판단.

## 7. 수렴 감지 — Ralph 스타일

### 개념

무한 루프를 방지하는 알고리즘. "N번 돌리고 끝"이 아니라 **점수와 개선폭을 보고 의미 있는 재시도만** 한다. 핵심 질문: "점수가 올라가고 있는가?"

### 구현

```typescript
// src/lib/scoring.ts — checkConvergence()
evalHistory = [Round1 결과, Round2 결과, Round3 결과, ...]
```

### 3가지 판단 + 안전장치

```
방향 전환:  점수가 너무 낮음 → pivot (완전히 다른 접근으로 재시도)
계속:      점수 증가폭이 큼  → continue (잘 가고 있으니 유지)
멈춤:      점수 증가폭이 별 차이 없음 → accept (더 돌려도 제자리, 결과물은 가져감)

안전장치:  최대 라운드 수 (기본 3회) → 위 판단과 무관하게 강제 종료 (harness-config에서 설정)
```

**예시:**

```
[5.0]             → 너무 낮음 → pivot
[7.0, 7.8]        → Δ0.8 (큼) → continue
[7.0, 7.8, 8.1]   → Δ0.3 (적당) → continue → PASS (8.0 이상)
[7.0, 7.8, 7.9]   → Δ0.1 (미미) → accept (이 추세론 8.0 도달 어려움)
[7.5, 7.5, 7.5]   → Δ0 → accept
```

### 고민: 왜 패턴 분류(정체/진동/악화)를 안 하는가

이전에는 "정체인가? 진동인가? 악화인가?"를 분류했지만, 결국 핵심 질문은 "개선되고 있는가?"이다. 패턴 분류는 진단명을 붙이는 것이고, 점수/개선폭은 상태를 직접 측정하는 것. 분류하다 보면 "어디에도 안 걸리는" 케이스(감속 등)가 생기고, 직접 측정하면 모든 케이스를 커버한다.

## 8. 파일 시스템 — 에이전트 간 통신

### 개념

에이전트가 파일(md)로 소통. Planner가 spec.md를 쓰면 Generator가 읽고, Generator가 코드를 쓰면 Evaluator가 읽음.

### 구현: 파일 구조

```
.ai/harness/
├── {도메인}/
│   ├── domain-context.md          ← 도메인 공유 설계 (DTO, API, 타입)
│   ├── {페이지}/                  ← 실행 단위 = 페이지 1개
│   │   ├── spec.md                ← Planner 출력 (전체 스펙)
│   │   ├── sprint-1/
│   │   │   ├── contract.md        ← 합의 (보완 시 append)
│   │   │   ├── eval-log-r1.md     ← Round 1 평가 기록
│   │   │   ├── feedback-r1.md     ← Round 1 피드백 (FAIL 시)
│   │   │   └── eval-log-r2.md     ← Round 2 평가 기록
│   │   └── summary.md             ← 전체 실행 요약
```

### 각 파일의 역할


| 파일                | 누가 쓰나                     | 누가 읽나                | 역할                             |
| ----------------- | ------------------------- | -------------------- | ------------------------------ |
| spec.md           | Planner                   | Generator, Evaluator | 전체 스펙 (기능, UI, API, Sprint 계획) |
| contract.md       | Planner 초안 → Evaluator 검토 | Generator, Evaluator | Sprint별 완료 기준 + 범위 제한          |
| eval-log-rN.md    | Evaluator                 | Orchestrator (점수 파싱) | 4단계 평가 결과 + 점수                 |
| feedback-rN.md    | Evaluator                 | Generator            | 구체적 수정 지시 (FAIL 시)             |
| domain-context.md | Orchestrator              | Planner (다음 페이지)     | 도메인 공유 설계 (Sprint 완료 시 업데이트)   |
| summary.md        | Orchestrator              | 사람                   | Sprint별 점수, 라운드, 미해결 사항        |


### 정보 흐름도

```
Planner ──→ spec.md ──→ Generator (맥락), Evaluator (맥락)
Planner ──→ contract.md 초안
Evaluator ──→ contract.md 검토/보완
Generator ←── contract.md ──→ Evaluator
Generator ──→ [코드 파일] ──→ Evaluator
Evaluator ──→ eval-log-rN.md ──→ Orchestrator (점수 파싱)
Evaluator ──→ feedback-rN.md ──→ Generator (다음 라운드)

차단 (의도적):
  Generator ──✕──→ Evaluator의 추론 과정 (게이밍 방지)
  Evaluator ──✕──→ Generator의 사고 과정 (편향 방지)
```

### 고민: 왜 파일인가

- 사람이 중간 결과를 직접 열어볼 수 있음 (디버깅, 개입)
- 로그가 자동 축적됨 (eval-log가 라운드마다 쌓임)
- git 추적 가능

### 고민: 파일의 약점

코드(Orchestrator)가 eval-log에서 점수를 정규식으로 파싱 → Evaluator가 형식을 틀면 파싱 실패.
대안: json-schema로 코드용(JSON) + 사람용(md) 분리. 단, Claude Agent SDK가 json-schema를 공식 지원할 때 전환 (현재 미지원).

## 9. 모델 호출 — Claude Agent SDK

### 개념

코드에서 Claude를 호출하는 방법. 4가지가 있고 용도가 다름.

### 4가지 비교

```
Plugin (Agent 도구)    — CC 안에서. 코드 제어 불가. 구독.
claude -p (CLI)        — 프로세스 스폰. 1회용. 구독.
Claude Agent SDK       — async 스트리밍. 세션 resume. 구독. ← 채택
Anthropic SDK (API)    — 완전 제어. 도구 직접 구현. 토큰 종량.
```

### 구현

```typescript
// src/lib/claude.ts
import { query } from '@anthropic-ai/claude-agent-sdk';

export async function callClaude(prompt: string, options: ClaudeOptions = {}): Promise<string> {
  for await (const message of query({
    prompt,
    options: { model, permissionMode: 'bypassPermissions', systemPrompt },
  })) {
    if (msg.type === 'result') result = msg.result;
  }
  return result;
}
```

### 고민: 왜 Agent SDK인가

처음엔 `claude -p`(CLI)로 구현. 문제 발견 후 전환:

- `claude -p`: 매 호출마다 프로세스 스폰 오버헤드 + 멀티턴 불가
- Agent SDK: 스트리밍 + 세션 resume 가능 + `callClaude` 함수 하나만 교체

Anthropic SDK(API)는 탈락: Generator가 파일을 직접 써야 하는데 tool_use 핸들러를 전부 구현해야 함. Agent SDK는 Claude Code 도구(Read/Write/Bash)를 공짜로 씀.

상세: `docs/harness/2026-03-31-claude-호출-방식-비교.md`

## 10. 두 가지 구현 — Plugin vs 서비스

### 왜 둘 다 만드는가

**핵심 가설: "컨텍스트 격리가 품질에 영향을 주는가?"**

```
A (Plugin):  같은 세션 → Generator가 Evaluator 컨텍스트를 일부 볼 수 있음
B (서비스):  별도 프로세스 → 완전 격리 → 서로의 추론을 절대 못 봄
```

GAN 원리에서 정보 차단이 핵심. A는 차단이 불완전, B는 완전. 이 차이가 결과물 품질에 영향을 주는지 실험.

### 구현 비교


|              | A. Plugin              | B. 독립 서비스                      |
| ------------ | ---------------------- | ------------------------------ |
| 에이전트 정의      | `.md` 파일 (프롬프트)        | `.ts` 파일 (함수)                  |
| Orchestrator | 스킬 프롬프트 (Claude가 분기)   | TypeScript 코드 (if/else)        |
| 모델 호출        | Claude Code Agent 도구   | Claude Agent SDK               |
| 실행           | `/fe:harness` (CC 안에서) | `pnpm fe-harness "요구사항"` (터미널) |
| 격리           | 부분적                    | 완전                             |


### 공유하는 것

- 프롬프트 내용 (Planner/Generator/Evaluator 역할 정의)
- 파일 구조 (.ai/harness/)
- 점수 공식, 수렴 감지 로직
- Contract/eval-log/feedback 형식

## 11. Ouroboros에서 가져온 것 / 버린 것

참조: `docs/harness/2026-02-23-ouroboros-deep-analysis.md`

### 가져온 것: 루프 개선 (진화 루프)

기존: 코드만 고침 → 기준이 빠져있으면 영원히 못 잡음
개선: Evaluator D단계로 contract도 진화 → 근본 원인 해결

### 버린 것 3가지

**병렬 실행** — FE Sprint 간 의존성이 높아서 효과 제한적 (UI가 API 타입을 import)
**메모리 통신** — 프로세스가 분리되어 메모리 공유 안 됨. 파일이 자연스러운 인터페이스
**SQLite 영속화** — 분석할 데이터가 아직 없음. cc-monitor 연동 시점에 추가
